## §1. 락 키 vs owner — 가장 헷갈리는 구조

분산락의 두 요소를 정확히 구분해야 모든 게 보임.

```java
String lockKey = "lock:seat:7";              // 자원 단위 공유 키
String owner   = UUID.randomUUID().toString(); // 누가 잡았나 표식 (클라이언트마다 다름)

redis.setIfAbsent(lockKey, owner, 10, TimeUnit.SECONDS);
//                  ↑ NX                  ↑ TTL
```

### 차이점 정리

| 항목 | 클라이언트 A | 클라이언트 B |
|---|---|---|
| 락 키 | `"lock:seat:7"` | **`"lock:seat:7"`** (같음) |
| owner 값 | `"AAA-uuid"` | `"BBB-uuid"` (다름) |

**핵심**:
- **락 키 이름** = 자원(좌석 7번)에 묶임. 모든 클라이언트가 같은 키.
- **owner 값** = "내가 잡았다" 표식. 클라이언트마다 다름.

### ❌ 잘못된 설계

```java
// 클라이언트마다 다른 키 사용 — 락의 의미 없어짐
String lockKey = "lock:seat:7:" + clientId;
```
→ 모든 클라이언트가 각자 다른 키 점유 → **동시 진입 가능** → 락 무효.

### ✅ 올바른 설계

```java
// 자원 단위로 키 공유 — 동시성 제어 가능
String lockKey = "lock:seat:7";
String owner   = UUID.randomUUID().toString();   // owner로 식별
```

## §2. NX의 진짜 의미 — "그 순간"만 1명 보장

> NX = "지금 키가 없으면 set, 있으면 실패"
> "한 번 잡으면 영원히 내 거"는 **보장 X**.

```
NX 보장 ✅: 동시에 여럿이 같은 키 노릴 때 → 1명만 성공
NX 보장 ❌: 시간 지나서 TTL 만료 → 다른 사람이 잡는 것
```

### 시간축으로 보기

```
10:00:00  [A] setIfAbsent("lock:7", "AAA", 10s) → ✅ 성공
          Redis: "lock:7" = "AAA" (TTL 10s)

10:00:05  [B] setIfAbsent → ❌ 실패 (NX 작동, A가 점유)

10:00:10  ⏰ TTL 만료 → Redis가 키 자동 삭제

10:00:11  [B] setIfAbsent → ✅ 성공 (키 비었으니 NX 통과)
          Redis: "lock:7" = "BBB" (TTL 10s)

10:00:15  [A] (작업 15초 걸려서 이제 끝남)
          redis.del("lock:7") 호출 → 💥 B의 락 삭제됨!
```

→ A·B는 "동시"에 락 가진 적 없음 (NX는 잘 지켜짐).
→ **순차적**으로 주인이 바뀌었을 뿐. A가 그걸 모르고 DEL하는 게 문제.

### 두 안전장치의 역할 분담

| 장치 | 보호하는 것 |
|---|---|
| **NX** | 동시 다중 점유 방지 (같은 순간의 보호) |
| **TTL** | 데드락 방지 (클라이언트 죽어도 락 영원히 안 풀리는 사태 방지) |
| **owner 검증 (Lua)** | 시간 흐른 후 남의 락 삭제 방지 (순차성의 보호) |

→ **세 가지가 함께** 분산락의 안전성을 만든다.

## §3. Stale Lock Release — 가장 유명한 함정

### 공식 명칭들

| 명칭 | 한국어 | 사용 컨텍스트 |
|---|---|---|
| **Stale Lock Release** | 만료된 락 잘못 해제 | 🎯 가장 정확 |
| **Lock Expiration Race** | 락 만료 경쟁 상태 | 학술적 |
| **Lock Hijacking / Stealing** | 락 탈취 | 의도성 뉘앙스 |
| **Cross-Client Lock Deletion** | 클라이언트 교차 삭제 | 결과 중심 |
| **GC Pause Problem** | GC 정지 문제 | Kleppmann 컨텍스트 |

### 시나리오 (다시 정리)

```
1. A 락 획득 (TTL 10초)
2. A 작업이 15초 걸림 (DB 느림 / GC pause / 네트워크 지연)
3. TTL 만료 → 락 자동 해제
4. B 락 획득
5. A가 작업 끝나고 DEL → 💥 B의 락 삭제
6. C 락 획득 → B, C 동시 임계 영역 진입 💥
```

### 해결책 1: Owner-aware Release (Lua) — 우리 패턴

```lua
-- KEYS[1] = 락 키, ARGV[1] = 내 owner UUID
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
```

```java
DefaultRedisScript<Long> releaseScript = new DefaultRedisScript<>(luaScript, Long.class);
redis.execute(releaseScript, List.of(lockKey), owner);
```

→ **"내가 잡은 락 맞을 때만 삭제"** 가 원자적으로 보장.
→ Redisson `unlock()`이 내부적으로 이걸 함.

### 해결책 2: Fencing Token (Kleppmann 제안)

```
락 획득할 때마다 단조 증가하는 숫자(토큰)를 함께 발급:

A: 락 획득 → token=33
A: GC pause...
B: 락 획득 → token=34   (반드시 더 큰 숫자)
A: 깨어남 → 작업 시도 (token=33)
저장소: "최신 토큰은 34인데, 33은 과거 → 거부" ✅
```

→ **이미 지나간 토큰은 거절**하는 안전장치.
→ 단, 저장소(DB)도 토큰을 인식해야 함 → 구현 복잡.
→ Redis 단독으론 어려움. 보통 **owner 검증(Lua)** 으로 충분.

## §4. Watchdog 🐕 — TTL 딜레마의 진짜 해결책

### TTL 설정의 딜레마

TTL을 어떻게 설정해도 한쪽이 깨짐:

| TTL 설정 | 문제 |
|---|---|
| **TTL 없음** | 클라이언트 죽으면 락 영원히 안 풀림 → 데드락 💀 |
| **TTL 짧게 (10초)** | 작업이 11초 걸리면 만료 → 다른 사람이 잡음 → 충돌 💥 |
| **TTL 길게 (10분)** | 클라이언트 죽으면 다른 요청들이 10분 대기 😴 |

→ 정적인 TTL로는 해결 불가. **동적으로 갱신해야 함.**

### Watchdog의 핵심 아이디어

> **"작업 중이면 TTL을 자동으로 계속 갱신해줘"**

```
락 잡을 때: TTL = 30초
10초마다: 백그라운드 스레드가 "아직 작업 중?" → 예 → TTL 30초로 리셋
20초마다: 또 체크 → 예 → 리셋
...
작업 끝남: 더 이상 갱신 안 함 → 명시적 unlock으로 즉시 해제
또는 30초 후 자동 해제 (안전망)

클라이언트 죽음: Watchdog도 같이 죽음 → TTL 갱신 멈춤 → 30초 후 자동 해제 ✅
```

### 시간축 비교

#### ❌ Watchdog 없으면

```
10:00:00  A 락 획득 (TTL 10초)
10:00:10  ⏰ TTL 만료 → 자동 해제
10:00:11  B 락 획득 → 동시 작업 💥
10:00:30  A 작업 완료 (이미 사고남)
```

#### ✅ Watchdog 있으면 (Redisson)

```
10:00:00  A 락 획득 (TTL 30초)
10:00:10  🐕 Watchdog: "아직?" → 예 → TTL 30초 재설정
10:00:20  🐕 "아직?" → 예 → 재설정
10:00:30  🐕 "아직?" → 예 → 재설정
10:00:50  A 작업 완료 → unlock() → 즉시 해제 ✅
   (절대 만료 안 됨)
```

#### ✅ 클라이언트 죽어도 안전

```
10:00:00  A 락 획득 (TTL 30초)
10:00:10  🐕 Watchdog 연장
10:00:20  💀 A 프로세스 죽음 (서버 다운, OOM 등)
          → Watchdog도 같이 죽음 → TTL 갱신 멈춤
10:00:50  ⏰ TTL 만료 → 정상 자동 해제 ✅
10:00:51  B 정상 진입
```

→ **작업 중이면 절대 만료 X, 죽으면 정상 회수**. 두 마리 토끼 다 잡음.

### 비유 — 화장실 키 + "사용 중" 버튼 🚻

```
규칙: 키 받은 후 10분 안에 반납 안 하면 자동 회수 (TTL)

[문제]
A가 키 받고 화장실 들어감 → 일이 15분 걸림
→ 10분 후 자동 회수 → B가 키 받음 → 같은 화장실로 💥

[해결: Watchdog = 화장실 안의 "사용 중" 버튼 🔔]
A가 5분마다 버튼 누름 → 회수 타이머 리셋
A가 정상 마무리하고 키 반납 → 즉시 해제
A가 쓰러져 버튼 못 누름 → 5분 후 자동 회수 → 직원이 구함 ✅

핵심: "살아서 작업 중이면 절대 회수 안 함, 죽으면 안전하게 회수"
```

### Redisson 코드 — `lock()` vs `tryLock(대기, 점유, 단위)`

#### `lock()` — Watchdog 자동 ON ⭐

```java
RLock lock = redisson.getLock("lock:seat:7");
lock.lock();   // 🐕 Watchdog 자동 시작 (TTL 30초 + 10초마다 갱신)
try {
    // 작업 (얼마나 걸리든 안전)
    paymentService.process(...);
} finally {
    lock.unlock();   // 즉시 해제 + Watchdog 중지
}
```

#### `tryLock(waitTime, leaseTime, unit)` — Watchdog OFF

```java
lock.tryLock(3, 10, TimeUnit.SECONDS);
// 3초 대기, 10초 점유. 10초 후엔 무조건 해제 (Watchdog 안 함)
```

→ **leaseTime을 명시하면 Watchdog 비활성화**. 정확히 N초만 점유.

#### 어떤 걸 써야 할까?

| 상황 | 추천 |
|---|---|
| 작업 시간 불확실 (외부 API, DB 느림) | `lock()` (Watchdog) ⭐ |
| 작업 시간 짧고 예측 가능 (1-2초) | `tryLock(대기, 점유, ...)` |
| 절대 N초 넘으면 안 됨 (강제 타임아웃) | `tryLock(대기, 점유, ...)` |

→ **일반적으론 `lock()` 추천.** leaseTime 짧게 줬다가 작업이 길어지면 §3 사고 발생.
→ **실무에선 사실상 `lock()` 한 줄로 끝.**

### 직접 구현하면? (학습용)

```java
String lockKey = "lock:seat:7";
String owner = UUID.randomUUID().toString();

if (redis.setIfAbsent(lockKey, owner, 30, TimeUnit.SECONDS)) {
    // Watchdog 시작 — 10초마다 TTL 갱신
    ScheduledFuture<?> watchdog = scheduler.scheduleAtFixedRate(() -> {
        redis.execute(extendScript, List.of(lockKey), owner, "30");
        // Lua로 owner 일치할 때만 갱신
    }, 10, 10, TimeUnit.SECONDS);

    try {
        // 작업
    } finally {
        watchdog.cancel(false);   // Watchdog 중지
        redis.execute(releaseScript, List.of(lockKey), owner);   // 안전 해제
    }
}
```

→ 직접 짜면 복잡하고 버그 잘 남. **그래서 Redisson `lock()` 한 줄이 정답.**

### 분산락의 4중 안전망 (완성)

```
┌─────────────────────────────────────────────────────┐
│  분산락의 4중 안전 메커니즘                            │
├─────────────────────────────────────────────────────┤
│  ① NX             → 같은 순간 1명만 (동시성)         │
│  ② TTL            → 데드락 방지 (장애 시 자동 해제)   │
│  ③ Watchdog 🐕    → 작업 중 TTL 자동 연장            │
│  ④ owner 검증     → 시간 흐른 후 남의 락 삭제 방지   │
│                                                     │
│  ↓ Redisson `lock()` 한 줄로 모두 자동 처리          │
└─────────────────────────────────────────────────────┘
```

| 안전망 | 보호하는 것 |
|---|---|
| **NX** | 동시 다중 점유 |
| **TTL** | 데드락 (클라이언트 죽었을 때) |
| **Watchdog** | **작업 도중 만료** |
| **owner 검증** | Stale Lock Release |

## §5. Martin Kleppmann의 Redlock 논쟁

분산락 학습 시 자주 등장하는 유명한 논쟁.

### 발단 — Kleppmann의 비판 (2016)

> **"How to do distributed locking"** — Martin Kleppmann
> `Designing Data-Intensive Applications` 저자

핵심 주장:
1. Redis의 **Redlock 알고리즘**(여러 Redis 인스턴스에 동시 락)은 안전하지 않다.
2. **GC pause**나 네트워크 지연으로 TTL 의존 락은 깨질 수 있음.
3. 해결책은 **Fencing Token** 같은 단조 증가 ID.

### 반박 — Antirez(Redis 창시자)

> **"Is Redlock safe?"** — Salvatore Sanfilippo
> Kleppmann의 가정이 비현실적이라고 반박. 실무에선 Redlock 충분히 안전.

### 결론 (실무 관점)

| 케이스 | 추천 |
|---|---|
| 일반 서비스 (대부분) | Redis 단일 인스턴스 + **Redisson** RLock |
| 안전성 절대 필요 (금융 등) | Fencing Token + DB 토큰 검증 |
| Redlock | 거의 안 씀. 복잡도 대비 이득 작음 |

→ **실무 99%는 Redisson `RLock` 으로 충분**.
→ 면접에서 깊이 들어가면 Kleppmann 논쟁 언급하면 좋음.

## §6. 락 vs DB — 역할 분업

> **락은 데이터 저장소가 아니다.** "지금 작업 중"이라는 일회성 신호일 뿐.
> 실제 상태(예매됨/취소됨 등)는 DB에 영구 저장.

### 두 메커니즘의 분업

| 보호 대상 | 도구 | 시간 범위 |
|---|---|---|
| **이미 예매된 좌석 막기** | DB (`status=BOOKED`) | 영구 |
| **동시 예매 충돌 막기** | 락 (Redis) | 작업 중 짧은 순간 |

### 흐름

```
사용자 예매 시도
  ↓
[락] 동시 진입 방지 ← 짧은 순간 (수 ms ~ 수 초)
  ↓
[DB] 좌석 상태 검사 ← 영구 상태 확인
  ↓
[DB] 좌석 상태 변경 ← 영구 기록
  ↓
[락] 해제 ← 다음 작업자 위해
```

### 왜 작업 끝나면 락을 해제해야 하나?

**락을 안 지우면**:
1. **다른 작업 막힘** — 환불, 취소, 좌석 정보 수정 등 모두 대기
2. **TTL 만료까지 무작정 대기** — 응답 시간 폭증
3. **잘못된 키 설계 시 다른 좌석도 막힘** — 락 키 범위 잘못 잡으면 더 큼

→ 락은 **작업 끝나면 즉시 해제**가 정답.

### 비유

```
🚻 화장실 키 (락) vs 사용 기록부 (DB)
🚦 도로 신호등 (락) vs CCTV 영상 (DB)

키/신호등 = 일시적 통제 → 끝나면 회수/꺼야 함
기록부/CCTV = 영구 보관 → 영원히 남김
```

## §7. Check-Then-Act 함정 — 락이 왜 필요한가

락 없이 "조회 → 검사 → 행동" 하면 발생하는 동시성 문제.

### ❌ 락 없이

```java
public void bookSeat(Long seatId, Long userId) {
    Seat seat = seatRepository.findById(seatId);  // ① 조회
    if (seat.isBooked()) throw new AlreadyBookedException();  // ② 검사
    seat.book(userId);                              // ③ 행동
    seatRepository.save(seat);                      // ④ 저장
}
```

```
[좌석 7번 비어있는 상태에서 A, B 동시 클릭]

A: ① 조회 → "비어있음"
B: ① 조회 → "비어있음"     ← A와 동시!
A: ② 검사 → 통과
B: ② 검사 → 통과
A: ③④ 예매 → seat=BOOKED
B: ③④ 예매 → seat=BOOKED (덮어쓰기)

→ 한 좌석 두 명 예매 (oversell) 💥
```

**문제**: DB가 막아줄 거라 믿었는데, **"조회 → 검사 → 저장" 사이의 짧은 틈**에 다른 요청이 끼어듦.

### ✅ 락으로 해결

```java
public void bookSeat(Long seatId, Long userId) {
    RLock lock = redisson.getLock("lock:seat:" + seatId);
    lock.tryLock(3, 10, TimeUnit.SECONDS);
    try {
        // 락 안에서는 한 명만 들어옴 = 전체 시퀀스가 안전
        Seat seat = seatRepository.findById(seatId);
        if (seat.isBooked()) throw new AlreadyBookedException();
        seat.book(userId);
        seatRepository.save(seat);
    } finally {
        lock.unlock();
    }
}
```

```
A: 락 획득 → 안전한 시퀀스 → 락 해제
B: A 끝날 때까지 대기 → 락 획득 → "이미 BOOKED" → 차단 ✅
C: 좌석 8번 예매 → "lock:seat:8"은 다른 키 → 자유 ✅
```

### 대안: DB 비관적 락 / 낙관적 락

```java
// 비관적 락 (SELECT FOR UPDATE)
@Lock(LockModeType.PESSIMISTIC_WRITE)
Seat findById(Long id);

// 낙관적 락 (@Version)
@Version
private Long version;
```

→ **DB 락**도 가능. 단, 단일 DB일 때만. **분산 환경**(여러 서비스, 외부 API 호출 등)에선 Redis 분산락이 더 유연.

## §8. 흔한 함정

| 함정 | 결과 | 해법 |
|---|---|---|
| `SETNX` + `EXPIRE` 분리 호출 | 두 명령 사이 죽으면 데드락 | `SET k v NX EX 10` 원자적 |
| `DEL`만 호출 (owner 검증 X) | Stale Lock Release | Lua + owner 검증 |
| TTL 너무 짧음 | 작업 끝나기 전 만료 → 동시 진입 | Watchdog (Redisson) 또는 충분히 길게 |
| TTL 너무 김 | 클라이언트 죽으면 다른 요청 다 막힘 | 적당히 (작업 예상 × 2~3배) |
| 재진입(reentrant) 불가 | 같은 클라이언트도 재진입 못 함 | Redisson RLock 사용 |
| 락 안에서 외부 호출 (HTTP, 외부 PG) | 작업 시간 길어져 락 만료 | 락 밖으로 빼거나 Watchdog |
| 락 키와 데이터 키 혼동 | 락 안 풀어서 다른 작업 막힘 | 락은 신호, 데이터는 DB |
| 클라이언트별 락 키 다름 | 동시성 제어 불가 | 자원 단위 키 공유 |

## §9. 이 프로젝트 적용 포인트

### 1. 좌석 예매 (향후 추가 시)

```java
public void bookSeat(Long concertId, String seatNo, Long userId) {
    RLock lock = redisson.getLock("lock:seat:" + concertId + ":" + seatNo);
    boolean acquired = false;
    try {
        acquired = lock.tryLock(3, 10, TimeUnit.SECONDS);
        if (!acquired) throw new CustomException(ErrorCode.SEAT_LOCKED);

        Seat seat = seatRepository.findByConcertAndNo(concertId, seatNo);
        if (seat.isBooked()) throw new CustomException(ALREADY_BOOKED);
        seat.book(userId);
        seatRepository.save(seat);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        throw new CustomException(LOCK_INTERRUPTED);
    } finally {
        if (acquired && lock.isHeldByCurrentThread()) lock.unlock();
    }
}
```

→ 부록-멱등성.md와 결합.

### 4. 좌석 재고 카운터 차감

```java
// 음수 방어 Lua 스크립트와 결합
RLock lock = redisson.getLock("lock:stock:" + concertId);
lock.tryLock(...);

    Long left = redis.execute(decrStockScript, List.of("stock:" + concertId));
```

## §10. Redisson과의 관계

이 부록의 모든 패턴은 **Redisson `RLock`이 자동 처리**:

| 패턴 | 직접 구현 | Redisson |
|---|---|---|
| owner UUID 생성 | 수동 | 자동 |
| Lua로 안전 해제 | 직접 작성 | 자동 |
| 작업 중 TTL 갱신 (Watchdog) | 별도 스레드 | 자동 |
| 락 대기 (Pub/Sub) | 폴링 | 자동 |
| 재진입 지원 | 추가 구조 필요 | 자동 |

→ **실무 신규 프로젝트는 처음부터 Redisson 사용이 표준.**
→ 직접 구현은 학습용으로만.

자세한 Redisson 문법은 **02-실무.md §5 "Redisson 실전 사용법"** 참고.

## §11. 한 줄 정리

> **분산락 = "자원에 대한 동시 수정을 한 명만 허용" 🔒**
>
> **4중 안전망**:
> - **NX** = 같은 순간 보호 (동시성)
> - **TTL** = 데드락 방지 (장애 시 자동 해제)
> - **Watchdog 🐕** = 작업 도중 TTL 자동 연장 (Redisson)
> - **owner 검증 (Lua)** = Stale Lock Release 방지
>
> 락 키는 **자원 단위로 공유**(`"lock:seat:7"`), **owner는 UUID로 식별**.
>
> 락은 **신호등**이지 **데이터 저장소** 아님. 작업 끝나면 즉시 해제.
> 실제 상태는 **DB에 영구 저장**.
>
> **실무는 사실상 Redisson `lock.lock()` 한 줄로 끝.** 위 4중 안전망 다 자동 처리.
> 직접 구현은 학습용으로만.

## 외울 템플릿

### 직접 구현 (학습용)

```java
String lockKey = "lock:" + resource;
String owner = UUID.randomUUID().toString();

if (redis.setIfAbsent(lockKey, owner, 10, TimeUnit.SECONDS)) {
    try {
        // 작업
    } finally {
        // Lua로 owner 검증 후 해제
        redis.execute(releaseScript, List.of(lockKey), owner);
    }
}
```

### Redisson (실무)

```java
RLock lock = redisson.getLock("lock:" + resource);
if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
    try {
        // 작업
    } finally {
        if (lock.isHeldByCurrentThread()) lock.unlock();
    }
}
```
