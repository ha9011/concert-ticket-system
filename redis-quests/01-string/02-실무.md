# String 실무 — 5대 핵심 패턴 + 이 프로젝트 적용 가이드

> 명령어를 안다 ≠ 실무에서 쓸 줄 안다. 이 문서는 **"실제 서비스가 마주치는 문제 → Redis로 어떻게 해결하는가"**에 집중.
> 모든 예제는 이 프로젝트(`concert-ticket-system/backend`)의 실제 코드와 엮음.

## 다루는 5가지

1. **Cache-Aside** — 가장 흔한 캐싱 패턴
2. **인증 코드** — TTL + 검증 + 시도 횟수 제한
3. **멱등성 키 (Idempotency Key)** — 중복 요청 차단
4. **카운터** — 조회수, Rate Limit, 재고
5. **분산락** — 좌석 예매, 결제 동시성 제어

---

# §1. Cache-Aside 패턴

## 문제 상황

```
1. API 요청 → DB 조회 → 응답 (느림: DB 부담↑, 응답 시간↑)
2. 같은 데이터를 1초에 1000번 요청받으면? DB가 비명을 지른다.
```

## 표준 패턴 (Cache-Aside, "Lazy Loading")

```
[조회 흐름]
1. Redis에서 캐시 조회 (GET)
2. 캐시 HIT → 그대로 반환
3. 캐시 MISS → DB 조회 → Redis에 저장 (SET + TTL) → 반환

[갱신 흐름]
1. DB 업데이트
2. Redis 캐시 무효화 (DEL)
3. 다음 조회 때 다시 캐싱됨
```

### 코드 형태 (수동 구현)

```java
public User getUser(Long userId) {
    String key = "user:" + userId;

    // 1. 캐시 조회
    String cached = redisTemplate.opsForValue().get(key);
    if (cached != null) {
        return objectMapper.readValue(cached, User.class);   // HIT
    }

    // 2. DB 조회
    User user = userRepository.findById(userId).orElseThrow();

    // 3. 캐시에 저장 (TTL 5분)
    redisTemplate.opsForValue().set(key, objectMapper.writeValueAsString(user), 5, TimeUnit.MINUTES);

    return user;
}

public void updateUser(Long userId, ...) {
    userRepository.save(user);
    redisTemplate.delete("user:" + userId);   // 캐시 무효화
}
```

### Spring Cache 애너테이션 (선언적, 권장)

```java
@Cacheable(value = "user", key = "#userId")
public User getUser(Long userId) {
    return userRepository.findById(userId).orElseThrow();
}

@CacheEvict(value = "user", key = "#userId")
public void updateUser(Long userId, ...) { ... }
```

## 흔한 함정 5가지

### 1. 캐시 무효화 누락

```java
// ❌ DB만 업데이트, 캐시는 그대로 → "유령 데이터" 노출
userRepository.save(user);
```

**해법**: 업데이트 메서드에 `@CacheEvict` 또는 `redisTemplate.delete()` 반드시 동반.

### 2. TTL 미설정 → 메모리 누수

```java
redisTemplate.opsForValue().set(key, value);   // ❌ 영원히 남음
redisTemplate.opsForValue().set(key, value, 5, TimeUnit.MINUTES);   // ✅
```

### 3. SET 덮어쓸 때 TTL 날아감 (§4 함정 재등장)

```java
// 캐시 갱신하려고 SET 다시 호출했더니 TTL이 -1 (영구) 됨
redisTemplate.opsForValue().set(key, newValue);   // ❌ TTL 사라짐

// 해법
redisTemplate.execute((RedisCallback<Boolean>) c ->
    c.set(key.getBytes(), newValue.getBytes(),
          Expiration.keepTtl(), RedisStringCommands.SetOption.UPSERT));
// 또는 그냥 매번 TTL 같이 주기 (가장 흔한 방식)
redisTemplate.opsForValue().set(key, newValue, 5, TimeUnit.MINUTES);
```

### 4. 캐시 스탬피드 (Cache Stampede)

**상황**: 인기 캐시가 동시에 만료되면, 수천 개 요청이 한꺼번에 DB로 몰려 DB가 죽음.

**해법**:
- TTL에 jitter 추가: `5분 ± 30초 랜덤`
- 분산락으로 "DB 조회는 한 명만" 강제 (이 문서 §5)
- PER (Probabilistic Early Recomputation): TTL 끝나기 직전 일부 요청만 미리 갱신

### 5. Null 캐싱 함정

```java
// DB에 없는 ID 조회 → null 반환 → 캐싱 안 함 → 매번 DB 조회 (DB 부담)
// 해법: "없음" 자체를 짧은 TTL로 캐싱
redisTemplate.opsForValue().set("user:999", "__NOT_FOUND__", 30, TimeUnit.SECONDS);
```

## 🎯 이 프로젝트 적용 포인트

현재 프로젝트는 **Cache-Aside 미구현** (`@Cacheable` 사용 없음). 적용해볼 만한 곳:

| 대상 | 키 패턴 | TTL | 이유 |
|---|---|---|---|
| `User` 조회 | `user:{id}` | 10분 | 로그인/세션 검증 시 빈번 조회 |
| `EmailLog` 집계 (사용자별 발송 횟수) | `emaillog:count:{userId}` | 1분 | 통계 조회 시 매번 COUNT 쿼리 회피 |

**참고 코드**: `api-server/src/test/java/com/concert/api/integration/RedisFuturePatternTest.java:64-97` 에 수동 Cache-Aside 패턴 테스트가 이미 있음 → 학습 레퍼런스로 활용.

# §2. 인증 코드 패턴

## 문제 상황

이메일/SMS 인증 코드를 어떻게 안전하게 관리할까?
- 발급 → TTL 안에 사용자가 입력 → 검증 → 즉시 폐기
- 무차별 대입 공격(브루트포스) 방어
- 같은 이메일에 코드 도배 방지

## 표준 패턴 (4단계)

### Step 1: 코드 발급 (Rate Limit + TTL)

```java
String code = String.format("%04d", new Random().nextInt(10000));

// NX 옵션: 유효한 코드가 이미 있으면 새로 발급 안 함 (도배 방지)
Boolean ok = redisTemplate.opsForValue()
    .setIfAbsent("auth:" + email, code, 3, TimeUnit.MINUTES);

if (!ok) {
    throw new CustomException(ErrorCode.AUTH_CODE_ALREADY_ISSUED);
}
// 이메일 발송
```

**핵심**: `NX + EX 180` 조합 → "기존 코드 만료 전엔 재발급 불가" 자동 보장.

### Step 2: 검증 (compare-and-delete + 시도 횟수)

❌ **잘못된 패턴**: 단순 GET + DEL
```java
String saved = redis.get("auth:" + email);
if (!saved.equals(input)) return fail();
redis.delete("auth:" + email);   // 일치 시에만 삭제
```
→ GET-DEL 사이에 다른 요청이 끼어들 수 있음 (race condition).
→ 또한 사용자가 1234를 1234567번 시도해도 막을 방법이 없음.

✅ **올바른 패턴**: Lua 스크립트로 원자화

```lua
-- KEYS[1] = "auth:user@x.com"
-- KEYS[2] = "auth:attempts:user@x.com"
-- ARGV[1] = 사용자 입력
-- ARGV[2] = 최대 시도 횟수 ("5")

local stored = redis.call('GET', KEYS[1])
if not stored then
    return 'EXPIRED'
end

if stored == ARGV[1] then
    redis.call('DEL', KEYS[1])
    redis.call('DEL', KEYS[2])
    return 'OK'
end

local attempts = redis.call('INCR', KEYS[2])
redis.call('EXPIRE', KEYS[2], 180)

if tonumber(attempts) >= tonumber(ARGV[2]) then
    redis.call('DEL', KEYS[1])
    redis.call('DEL', KEYS[2])
    return 'BLOCKED'
end

return 'MISMATCH'
```

```java
DefaultRedisScript<String> script = new DefaultRedisScript<>(luaScript, String.class);
String result = redisTemplate.execute(
    script,
    List.of("auth:" + email, "auth:attempts:" + email),
    userInput, "5"
);

switch (result) {
    case "OK"       -> /* 인증 성공 */
    case "MISMATCH" -> /* 코드 불일치, 재시도 가능 */
    case "BLOCKED"  -> /* 5회 초과, 코드 무효화 */
    case "EXPIRED"  -> /* TTL 만료 */
}
```

### Step 3: 발송 Rate Limit (분당/시간당)

```java
String rlKey = "auth:rl:" + email;
Long count = redisTemplate.opsForValue().increment(rlKey);
if (count == 1) {
    redisTemplate.expire(rlKey, 1, TimeUnit.HOURS);
}
if (count > 5) {
    throw new CustomException(ErrorCode.TOO_MANY_AUTH_REQUESTS);
}
```

### Step 4: 회원가입 완료 → 폐기

검증이 OK면 위 Lua 스크립트가 자동으로 DEL 처리.

## 흔한 함정

| 함정 | 결과 |
|---|---|
| GET + DEL 분리 호출 | race condition으로 중복 가입 가능 |
| GETDEL 단독 사용 | 사용자가 오타 한 번 내면 코드 사라짐, 재발송 비용↑ |
| 시도 횟수 제한 없음 | 브루트포스로 4자리 코드 (10000개) 다 시도 가능 |
| NX 미사용 | 한 이메일에 코드 100개 발사 가능 (스팸 도구화) |
| TTL 안 줌 | 만료 안 되고 영원히 남음 |

## 🎯 이 프로젝트 적용 포인트

현재 `AuthService.java` 의 인증 코드 로직을 보면:

**현재 코드** (`api-server/src/main/java/com/concert/api/service/AuthService.java`):
```java
// L39 발급 - NX 옵션 없음 (도배 가능)
redisTemplate.opsForValue().set(AUTH_PREFIX + email, code, 3, TimeUnit.MINUTES);

// L66 검증 - GET 후 비교
String savedCode = redisTemplate.opsForValue().get(AUTH_PREFIX + request.getEmail());
if (savedCode == null || !savedCode.equals(request.getCode())) { ... }

// L82 삭제 - 별도 호출
redisTemplate.delete(AUTH_PREFIX + request.getEmail());
```

**개선할 점**:
1. **L39** — `set()` → `setIfAbsent()` (NX) 으로 변경, 같은 이메일 도배 방지
2. **L66~82** — GET + DEL 분리 → Lua 스크립트로 원자화
3. **시도 횟수 카운터 추가** — `auth:attempts:{email}` 도입
4. **발송 Rate Limit 추가** — `auth:rl:{email}`, 시간당 5회 제한

→ §2의 4단계를 적용하면 인증 보안이 한 단계 올라감.

# §3. 멱등성 키 (Idempotency Key)

## 문제 상황

**중복 요청**은 분산 시스템의 흔한 골칫거리:
- 네트워크 타임아웃 → 클라이언트가 재시도 → 같은 결제 2번 처리
- 메시지 큐 at-least-once 전달 → 같은 이메일 2번 발송
- 사용자가 "결제" 버튼 빠르게 두 번 클릭

목표: **"같은 요청이 N번 와도 결과는 1번만 처리"**

## 표준 패턴

### 패턴 A: SET NX EX (가장 흔함)

```java
String idemKey = "idem:payment:" + request.getIdempotencyKey();

Boolean isFirst = redisTemplate.opsForValue()
    .setIfAbsent(idemKey, "PROCESSING", 24, TimeUnit.HOURS);

if (!isFirst) {
    // 두 번째 이상 요청 → 무시 또는 기존 결과 조회
    String result = redisTemplate.opsForValue().get(idemKey);
    return getCachedResult(result);
}

try {
    PaymentResult result = processPayment(...);
    redisTemplate.opsForValue().set(idemKey, result.getId(), 24, TimeUnit.HOURS);
    return result;
} catch (Exception e) {
    redisTemplate.delete(idemKey);   // 실패 시 재시도 가능하게
    throw e;
}
```

**핵심 옵션 조합**: `NX EX 24h`
- `NX`: 첫 요청만 진입 통과
- `EX 24h`: 일정 시간 후 같은 키 재사용 가능 (메모리 정리)

### 패턴 B: 메시지 큐 컨슈머 멱등성

큐 컨슈머가 같은 메시지를 두 번 받을 수 있을 때 (at-least-once 큐 특성):

```java
public void processEmail(String email) {
    String dedupKey = "dedup:email:" + email + ":" + LocalDate.now();

    Boolean isFirst = redisTemplate.opsForValue()
        .setIfAbsent(dedupKey, "1", 1, TimeUnit.DAYS);

    if (!isFirst) {
        log.info("Skip duplicate: {}", email);
        return;
    }

    sendEmail(email);
    emailLogRepository.save(...);
}
```

## 흔한 함정

| 함정 | 영향 |
|---|---|
| 멱등성 키 클라이언트 생성 안 함 | 클라이언트가 키를 안 보내면 의미 없음 (UUID v4 요구) |
| 실패 시 키 삭제 안 함 | 처리 실패해도 두 번째 요청이 막힘 → 영구 차단 |
| TTL 너무 짧음 | 늦게 도착한 재시도 요청을 새 요청으로 오인 |
| 결과 캐싱 안 함 | "이미 처리됨"만 알려주고 응답 못 줌 → 클라이언트는 결과 모름 |

## 🎯 이 프로젝트 적용 포인트

**1. 이메일 큐 컨슈머에 적용** (`queue-server/src/main/java/com/concert/queue/worker/EmailMultiWorker.java`):

현재는 `rightPop` 으로 큐에서 꺼내는 즉시 처리만 함. 중복 멱등성 처리 없음.

```
[현재] queue → pop → 바로 처리 → DB save
[개선] queue → pop → dedup 키 SETNX → 첫 처리면 진행, 중복이면 skip
```

키 설계: `dedup:email:{email}:{date}` 또는 메시지에 UUID 박아서 `dedup:email:{uuid}`.

**2. 향후 결제 API에 적용**:

콘서트 티켓 결제 API가 생기면, 클라이언트가 헤더로 `Idempotency-Key: <uuid>` 보내고 서버는 `idem:payment:{uuid}` 로 SETNX.

→ Stripe, PayPal 같은 결제 게이트웨이가 표준으로 쓰는 패턴.

# §4. 카운터 패턴

## 문제 상황

조회수, 좋아요, Rate Limit, 재고 등 **"숫자가 정확하고 빠르게 증가/감소"** 가 필요한 곳.

DB로 하면? `UPDATE post SET views = views + 1` → 매 요청마다 DB 쓰기. 1초 1만 요청이면 DB가 죽음.

## 표준 패턴

### 패턴 A: 단순 조회수

```java
public Long incrementViews(Long postId) {
    return redisTemplate.opsForValue().increment("views:post:" + postId);
}

public Long getViews(Long postId) {
    String value = redisTemplate.opsForValue().get("views:post:" + postId);
    return value == null ? 0L : Long.parseLong(value);
}
```

→ DB는 1분/1시간에 한 번 배치로 동기화 (write-behind).

### 패턴 B: Rate Limit (Fixed Window)

```java
public boolean allowRequest(String userId) {
    String minute = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMddHHmm"));
    String key = "rl:" + userId + ":" + minute;

    Long count = redisTemplate.opsForValue().increment(key);
    if (count == 1) {
        redisTemplate.expire(key, 1, TimeUnit.MINUTES);
    }

    return count <= 60;   // 분당 60회
}
```

**키 이름에 시간을 박는 게 핵심**: 시간이 바뀌면 자동으로 새 윈도우 시작, 옛 키는 TTL로 자동 삭제.

### 패턴 C: 재고 차감 (음수 방어)

❌ 단순 DECR — 0 미만으로 떨어질 수 있음
```java
Long left = redisTemplate.opsForValue().decrement("stock:concert:1");
if (left < 0) { /* 환불 보상 처리? 너무 늦음 */ }
```

✅ Lua 스크립트로 "체크 후 차감"
```lua
local stock = tonumber(redis.call('GET', KEYS[1]))
if not stock or stock <= 0 then
    return -1   -- 품절
end
return redis.call('DECR', KEYS[1])
```

```java
Long result = redisTemplate.execute(decrScript, List.of("stock:concert:1"));
if (result == -1) throw new CustomException(SOLD_OUT);
```

### 패턴 D: Sliding Window Rate Limit (Sorted Set)

Fixed Window의 약점(경계 직전 폭주)을 보완. **Sorted Set 단계에서 다룸 — 일단 이름만 기억**.

## 흔한 함정

| 함정 | 결과 |
|---|---|
| EX 안 줌 | 시간 지난 카운터가 영원히 메모리 점유 |
| INCRBYFLOAT 잔액 관리 | 부동소수점 부정확 → 회계 오차 |
| DB 동기화 없이 Redis만 의존 | Redis 장애 시 데이터 손실 |
| 단순 DECR로 재고 차감 | 음수 발생 가능 (oversell) |

## 🎯 이 프로젝트 적용 포인트

| 대상 | 키 | 명령어 |
|---|---|---|
| 사용자별 인증 코드 발송 횟수 | `auth:rl:{email}` | INCR + EXPIRE 1h |
| 콘서트 좌석 재고 | `stock:concert:{id}` | Lua DECR (음수 방어) |
| 회원가입 일별 통계 | `signup:count:{date}` | INCR + EXPIREAT 30일 후 |
| 이메일 발송 분당 처리량 (queue-server) | `email:rate:{minute}` | INCR + EXPIRE 60s |

# §5. 분산락 (Distributed Lock)

## 문제 상황

분산 환경에서 **공유 자원에 대한 동시 접근 제어**.

대표 예시: 콘서트 좌석 예매.

```
좌석 A에 동시에 2명이 결제 요청 → 둘 다 성공 → 한 좌석 두 명 발급 (oversell)
```

자바 `synchronized`? — 한 JVM 안에서만 유효. 분산 서버(api-server 2대 이상)에선 무효.

## 표준 패턴

### Step 1: 락 획득 — SET NX EX

```java
String lockKey = "lock:seat:" + concertId + ":" + seatNo;
String owner = UUID.randomUUID().toString();   // 락 소유자 ID

Boolean acquired = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, owner, 5, TimeUnit.SECONDS);

if (!Boolean.TRUE.equals(acquired)) {
    throw new CustomException(ALREADY_LOCKED);
}
```

핵심:
- **NX**: 누군가 이미 잡은 락이면 실패
- **EX 5**: 락 잡은 클라이언트가 죽어도 5초 후 자동 해제 (데드락 방지)
- **owner = UUID**: 내가 잡은 락인지 식별

### Step 2: 임계 영역 작업

```java
try {
    // 좌석 점유 확인 + DB 업데이트
    Seat seat = seatRepository.findByConcertAndNo(concertId, seatNo);
    if (seat.isBooked()) throw new CustomException(ALREADY_BOOKED);
    seat.book(userId);
    seatRepository.save(seat);
}
```

### Step 3: 락 해제 — 반드시 "내 락"만 (Lua)

❌ **잘못된 해제**:
```java
redisTemplate.delete(lockKey);   // 다른 클라이언트가 잡은 락도 지움!
```

**시나리오**:
1. 클라이언트 A가 락 획득 (TTL 5초)
2. A의 작업이 6초 걸림 → TTL 만료로 락 자동 해제
3. 클라이언트 B가 락 획득
4. A가 늦게 작업 완료 → `DEL lockKey` 호출 → **B의 락이 지워짐** 💥

✅ **올바른 해제** (Lua로 owner 검증):

```lua
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
```

```java
DefaultRedisScript<Long> releaseScript = new DefaultRedisScript<>(luaScript, Long.class);
redisTemplate.execute(releaseScript, List.of(lockKey), owner);
```

## 흔한 함정

| 함정 | 결과 |
|---|---|
| `SETNX` + `EXPIRE` 분리 | 두 명령 사이에 죽으면 데드락 |
| `DEL`로만 해제 (owner 검증 안 함) | 다른 클라이언트 락 지움 |
| TTL 너무 짧음 | 작업 끝나기 전에 락 만료 → 동시 접근 발생 |
| TTL 너무 김 | 클라이언트 죽었을 때 다른 요청 다 막힘 |
| 락 안에서 또 락 (재진입 불가) | 같은 클라이언트도 재진입 못 함 → Redisson의 RLock이 해결 |
| 락 안에서 외부 호출 (DB 트랜잭션, 외부 API) | 락 잡고 5초인데 외부 API가 10초 걸리면 끝장 |

## 더 강한 락이 필요하면: Redisson

위 패턴은 "기본 분산락"이고, 실무에서 자주 쓰이는 건 **Redisson 라이브러리의 RLock**:

```java
RLock lock = redisson.getLock("lock:seat:1:A1");
boolean ok = lock.tryLock(3, 10, TimeUnit.SECONDS);   // 3초 대기, 10초 점유
try {
    if (ok) { /* 작업 */ }
} finally {
    if (ok) lock.unlock();
}
```

기능:
- 자동 owner 검증
- watchdog: 작업이 길어지면 TTL 자동 연장
- 재진입(reentrant) 지원
- pub/sub로 락 대기 효율화

→ 신규 프로젝트는 보통 Redisson 권장.

## Redlock 알고리즘?

여러 Redis 인스턴스에 동시에 락을 잡는 방식. 마틴 클렙만(Designing Data-Intensive Applications 저자)이 "안전하지 않다"고 비판하면서 유명해진 논쟁. **면접에서 들을 수 있지만, 실무 99%는 단일 인스턴스 + Redisson으로 충분.**

## 🎯 이 프로젝트 적용 포인트

콘서트 티켓 시스템에서 좌석 예매 로직이 추후 추가되면 **무조건 필요**:

```java
public void bookSeat(Long concertId, String seatNo, Long userId) {
    String lockKey = "lock:seat:" + concertId + ":" + seatNo;
    String owner = UUID.randomUUID().toString();

    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, owner, 10, TimeUnit.SECONDS);

    if (!Boolean.TRUE.equals(acquired)) {
        throw new CustomException(ErrorCode.SEAT_LOCKED);
    }

    try {
        // 좌석 예매 처리
    } finally {
        // Lua로 owner 검증 후 해제
        redisTemplate.execute(releaseScript, List.of(lockKey), owner);
    }
}
```

**참고 코드**: `api-server/src/test/java/com/concert/api/integration/RedisFuturePatternTest.java:106-144` 에 분산락 테스트가 이미 있음 → 학습 레퍼런스로 활용.

# §6. 다섯 패턴 종합 비교표

| 패턴 | 핵심 명령어 | TTL | NX | 키 예시 |
|---|---|---|---|---|
| **Cache-Aside** | `SET k v EX` + `GET` | 5~30분 | ❌ | `user:42` |
| **인증 코드 발급** | `SET k v NX EX` | 3분 | ✅ | `auth:user@x.com` |
| **인증 코드 검증** | Lua compare-and-delete | — | — | `auth:user@x.com` |
| **멱등성 키** | `SET k v NX EX` | 24시간 | ✅ | `idem:payment:{uuid}` |
| **카운터** | `INCR` + `EXPIRE` | 윈도우 단위 | ❌ | `rl:user:42:202605221430` |
| **분산락** | `SET k v NX EX` + Lua DEL | 5~10초 | ✅ | `lock:seat:1:A1` |

> 👀 **공통점**: `SET ... NX EX` 가 5개 중 3개에서 등장.
> Redis의 분산 시스템 활용은 "`NX EX` 조합을 어디에 쓸지"가 절반.
