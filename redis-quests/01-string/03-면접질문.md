## A. 기본 개념

### Q1. Redis가 뭐고 왜 쓰는가?

✅ **답**:
> **인메모리 key-value 저장소**. RAM에 데이터를 두기 때문에 디스크 기반 DB보다 수십~수백 배 빠름. 단순 캐시뿐 아니라 **자료구조(List/Set/Hash/Sorted Set)와 원자적 명령**을 제공해서 큐, 세션, 카운터, 분산락, Rate Limit 등 다양한 분산 시스템 패턴에 쓰임.

> 💡 꼬리질문 대비: "단순 캐시면 Memcached 쓰면 되는데 왜 Redis?" → Q4 참고.

---

### Q2. Redis가 빠른 이유는?

✅ **답** (3가지):

1. **인메모리** — RAM 접근은 디스크보다 10만 배 이상 빠름
2. **단일 스레드 + 이벤트 루프** — 락/컨텍스트 스위칭 비용 없음, 명령이 직렬화되어 안전
3. **효율적 자료구조 + C 구현** — String의 SDS, Hash의 listpack/hashtable 등 메모리 효율적인 내부 표현

> 💡 보너스 한마디: "병목은 CPU가 아니라 네트워크/메모리라서 단일 스레드로 충분"

---

### Q3. Redis 주요 용도 5가지를 말해보세요.

✅ **답**:

1. **캐시** — DB 부담 감소 (Cache-Aside 패턴)
2. **세션 저장소** — Spring Session Redis 등, 분산 서버 간 세션 공유
3. **메시지 큐** — List(LPUSH/RPOP), Pub/Sub, Stream
4. **카운터 / Rate Limit** — INCR + EXPIRE
5. **분산락** — `SET NX EX` + Lua

> 💡 이 프로젝트의 실제 용도: 인증 코드(`auth:{email}`), 이메일 큐(`mail:queue`), Spring Session(`concert:session:*`)

---

### Q4. Memcached vs Redis 차이는?

✅ **답**:

| | Memcached | Redis |
|---|---|---|
| 자료구조 | String만 | String/List/Set/Hash/ZSet/Stream 등 |
| 영속성 (Persistence) | 없음 | RDB/AOF로 디스크 저장 |
| 복제·고가용성 | 제한적 | Sentinel·Cluster 기본 지원 |
| 트랜잭션·Lua | 없음 | MULTI/EXEC, Lua 스크립트 |
| Pub/Sub | 없음 | 지원 |
| 스레드 | 멀티 스레드 | 단일 스레드 (네트워크 IO만 6+에서 멀티) |

→ **결론**: 단순 캐싱은 Memcached도 OK. 복잡한 자료구조·원자성·고가용성이 필요하면 Redis.

---

### Q5. Redis의 한계나 단점은?

✅ **답**:

1. **메모리 비용** — RAM 가격이 디스크보다 훨씬 비쌈. 대용량 데이터엔 부적합
2. **영속성 약점** — RDB는 주기적 스냅샷이라 사이 데이터 손실 가능, AOF도 fsync 정책에 따라 손실 가능
3. **단일 스레드의 양면** — 큰 키/긴 명령(KEYS *, Lua) 하나가 전체를 막음
4. **분산 트랜잭션 부재** — Cluster에서 키가 다른 슬롯에 있으면 MULTI/EXEC 불가
5. **메모리 단편화** — long-running 인스턴스는 단편화로 실제 사용량보다 메모리 더 점유

## B. 자료구조

### Q6. Redis 5대 자료구조와 각 용도는?

✅ **답**:

| 자료구조 | 대표 용도 |
|---|---|
| **String** | 캐시, 인증 코드, 카운터, 분산락 |
| **List** | 큐(FIFO), 스택(LIFO), 최근 N개 로그 |
| **Set** | 태그, 친구 목록, 중복 제거, 교집합 (공통 친구) |
| **Hash** | 객체 필드별 저장 (`user:42` → name/email/tier) |
| **Sorted Set** | 랭킹, 시간순 정렬, Sliding Window Rate Limit |

> 💡 추가: Stream(메시지 큐 고급), HyperLogLog(근사 카운팅), Geo, Bitmap

---

### Q7. String의 최대 크기는? 언제 String 대신 Hash?

✅ **답**:

- **String 최대 크기**: 512 MB (사실상 거의 안 닿음)
- **Hash 선택 기준**:
  - **객체의 일부 필드만 자주 조회/갱신** 한다면 Hash가 유리 (필드 단위 GET/SET)
  - 매번 전체 JSON을 String으로 SET → 큰 객체 갱신 시 비효율
  - 메모리 효율도 Hash가 좋음 (작은 Hash는 listpack 인코딩으로 압축 저장)

```
[String] SET user:42 '{"name":"ha","email":"x","tier":"gold"}'
         → email만 바꾸려 해도 전체 직렬화 + SET

[Hash]   HSET user:42 name "ha" email "x" tier "gold"
         HSET user:42 email "y"   ← 한 필드만 갱신 가능
```

---

### Q8. 같은 키에 다른 자료구조 명령 실행하면?

✅ **답**:

> **`WRONGTYPE Operation against a key holding the wrong kind of value`** 에러.

**Redis는 키 단위로 자료구조 타입이 고정됨.** String으로 만든 키에 LPUSH/SADD/ZADD 등은 모두 실패.

```
> SET k "string"
> LPUSH k "x"
(error) WRONGTYPE ...
```

> 💡 해결: 키 네이밍 컨벤션으로 사전 분리 (`user:42:name` String, `user:42:tags` Set).

## C. 명령어 비교

### Q9. `SET` vs `SETNX` vs `SETEX` 차이는?

✅ **답**:

| 명령 | 의미 |
|---|---|
| `SET k v` | 그냥 저장 (덮어쓰기) |
| `SETNX k v` | 키가 **없을** 때만 저장 (NX = Not eXists) |
| `SETEX k 30 v` | TTL 30초로 저장 (key, seconds, value 순서 주의) |

→ **현대는 모두 `SET k v [NX|XX] [EX|PX|EXAT|PXAT] [GET] [KEEPTTL]` 하나로 통합됨**. 옛 명령어들은 deprecated 흐름.

---

### Q10. `SET ... NX EX` vs `SETNX + EXPIRE` 차이는?

✅ **답**:

> **원자성**의 차이.
>
> - `SETNX` + `EXPIRE`는 **두 명령**. 사이에 클라이언트가 죽으면 **TTL 없는 키가 영원히 남음 → 데드락**.
> - `SET k v NX EX 5`는 **한 명령**. Redis가 원자적으로 처리 → 절대 쪼개지지 않음.

→ Redis 2.6.12부터 `SET ... NX EX` 등장, 분산락의 표준이 됨.

> 💡 시나리오 1 참고: 인증 코드 발급에도 `NX EX` 조합이 도배 방지에 쓰임.

---

### Q11. `GETSET` vs `SET ... GET` 차이는?

✅ **답**:

- 둘 다 **이전 값을 반환하면서 새 값 저장** (atomic swap).
- `GETSET`은 Redis 6.2에서 **deprecated**. `SET ... GET`이 표준.
- 이유: `SET ... GET`은 **다른 옵션(NX/XX/EX/KEEPTTL)과 조합 가능**, `GETSET`은 안 됨.

```bash
# 옛날
> GETSET counter "0"

# 지금
> SET counter "0" GET EX 3600   ← TTL까지 한 방
```

> 💡 패턴 자체(atomic swap)는 카운터 리셋·Rate Limit 윈도우 교체에 여전히 자주 쓰임.

---

### Q12. `GET + DEL` vs `GETDEL` 차이는?

✅ **답**:

- `GET` + `DEL`은 **두 명령**. 사이에 다른 요청이 끼어들어 같은 값을 두 번 읽어 처리할 수 있음 (race condition).
- `GETDEL`은 **한 명령**. 조회와 삭제가 원자적.

**용도**: 한 번만 쓰고 버리는 토큰 (URL의 비밀번호 재설정 토큰, OAuth nonce, 결제 idempotency 키).

> 💡 단, **사용자가 타이핑하는 OTP 코드엔 부적합** (오타 시 코드 사라짐). 그땐 Lua로 compare-and-delete 패턴.

---

### Q13. `DEL` vs `UNLINK` 차이는?

✅ **답**:

| | DEL | UNLINK |
|---|---|---|
| 방식 | **동기** 삭제 | **비동기** 삭제 (백그라운드) |
| big key 영향 | 삭제 동안 Redis 전체 블로킹 | 즉시 응답, 백그라운드 처리 |
| 권장 | 작은 키 | **큰 키 (List/Hash/Set 수만 개 원소 등)** |

→ **운영의 정석**: 큰 키 삭제는 `UNLINK`.

---

### Q14. `KEYS` vs `SCAN` 차이는?

✅ **답**:

| | KEYS | SCAN |
|---|---|---|
| 방식 | 한 번에 전체 스캔 | 커서 기반 점진적 스캔 |
| 블로킹 | **전체 Redis 블로킹** (키 100만 개면 수 초) | 비블로킹 |
| 운영 | **절대 금지** | 사용 가능 |

→ **운영에서 KEYS는 사실상 사고**. `redis.conf` 의 `rename-command KEYS ""` 로 비활성화 권장.

> 💡 단, 애플리케이션 핫패스에선 SCAN도 비추 — 인덱스(SET)를 따로 두는 게 정석.

---

### Q15. `EXPIRE` vs `EXPIREAT` 차이는?

✅ **답**:

- `EXPIRE k 60` — **상대** 시간 (지금부터 60초 후)
- `EXPIREAT k 9999999999` — **절대** 시각 (Unix 타임스탬프, 초 단위)

**언제 절대 시각을 쓸까?**
> "2026-06-01 00:00 정확히 만료" 같은 경우. 쿠폰 만료, 이벤트 종료, 티켓 오픈 시각 등.

`PEXPIRE` / `PEXPIREAT`은 밀리초 단위 버전.

---

### Q16. `INCR` vs `INCRBYFLOAT` 차이는? 잔액 관리엔 뭘 써야?

✅ **답**:

- `INCR`: 정수만. 소수/문자열엔 에러. 오버플로(Long.MAX_VALUE) 시 에러.
- `INCRBYFLOAT`: 부동소수점 가능. **IEEE 754 정확도 문제 있음** (0.1+0.1+0.1 ≠ 0.3).

**잔액/금액 관리**:
> ❌ `INCRBYFLOAT`로 1500.50원 누적 — 오차 누적되면 회계 사고
> ✅ **원 단위 정수로 환산**해서 `INCRBY` — 150050(전(錢) 단위) 식

→ 금융 데이터는 **항상 정수**.

## D. 동시성·원자성

### Q17. Redis는 왜 단일 스레드인가?

✅ **답** (3가지 이유):

1. **락이 필요 없음** — 명령이 직렬화되어 자동 원자성
2. **컨텍스트 스위칭 비용 0** — 멀티 스레드의 오버헤드 제거
3. **CPU 병목이 아님** — 인메모리라 메모리·네트워크가 병목, 단일 스레드로 충분

> 💡 Redis 6부터 **네트워크 I/O는 멀티 스레드** (`io-threads`). 명령 실행은 여전히 단일 스레드.

---

### Q18. 단일 스레드인데 어떻게 수만 동시 요청을 처리?

✅ **답**:

> **이벤트 루프 (epoll/kqueue 기반 비동기 I/O)**.
>
> - 클라이언트 연결마다 스레드를 만드는 게 아니라, **하나의 스레드가 여러 소켓을 동시에 감시**.
> - 데이터 도착한 소켓만 처리. 다른 소켓은 대기.
> - Node.js와 같은 모델.

→ "단일 스레드 = 한 번에 하나만 처리"가 아니라 **"한 번에 하나의 명령을 실행"**.

> 💡 BLPOP 같은 블로킹 명령도 클라이언트만 대기 시킴 — Redis 서버 자체는 안 멈춤.

---

### Q19. `MULTI / EXEC` 트랜잭션이 뭔가?

✅ **답**:

```
> MULTI
OK
> SET a 1
QUEUED
> SET b 2
QUEUED
> EXEC
1) OK
2) OK
```

- 명령을 **큐에 모았다가 EXEC로 한꺼번에 실행**.
- 실행 중에는 다른 클라이언트의 명령이 끼어들지 못함 (원자성).

**한계**:
- RDBMS 트랜잭션과 다름 — **롤백 없음**. 한 명령이 실패해도 나머지는 그대로 실행됨.
- 조건부 로직 불가 (값 보고 다음 명령 결정 등).

→ 조건부 원자 작업은 **Lua 스크립트**가 정답.

---

### Q20. Lua 스크립트와 MULTI/EXEC 트랜잭션 차이는?

✅ **답**:

| | MULTI/EXEC | Lua |
|---|---|---|
| 명령 그룹화 | ✅ | ✅ |
| 원자적 실행 | ✅ | ✅ |
| 조건부 로직 | ❌ | ✅ (if/else) |
| 중간 값 사용 | ❌ | ✅ (변수) |
| 클라이언트-서버 round trip | 큐잉 단계마다 | **1번 (스크립트 전송 + 실행)** |

→ **복잡한 원자 작업 = Lua가 기본**. compare-and-delete, 음수 방어 재고 차감 등.

---

### Q21. Redis 분산락의 표준 패턴은?

✅ **답**:

```
1. 락 획득:   SET lock:resource <uuid> NX EX 10
2. 임계 작업
3. 락 해제:   Lua로 owner 검증 후 DEL
```

**3가지 핵심**:
1. **`NX EX`**: 원자적 락 획득 + 데드락 방지 TTL
2. **UUID owner**: 누구 락인지 식별
3. **Lua DEL**: TTL 만료 후 다른 클라이언트 락 잡았을 때 내가 잘못 지우는 사고 방지

```lua
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else return 0 end
```

> 💡 실무에선 **Redisson 라이브러리의 RLock**이 watchdog(TTL 자동 연장), 재진입, pub/sub 대기까지 제공.

## E. 운영·장애

### Q22. RDB vs AOF 차이는?

✅ **답**:

| | RDB | AOF |
|---|---|---|
| 방식 | 주기적 **스냅샷** | 모든 쓰기 **명령 로그** |
| 파일 크기 | 작음 (압축된 이진) | 큼 (텍스트 로그) |
| 복구 속도 | 빠름 | 느림 (명령 재실행) |
| 데이터 손실 | 스냅샷 간격만큼 가능 | fsync 정책에 따라 거의 없음 |
| 운영 비용 | 낮음 | 디스크 I/O 부담 |

**운영 권장**: 둘 다 켜기 (RDB는 백업·복구용, AOF는 최근 데이터 보호용).

---

### Q23. Redis maxmemory 도달 시 eviction 정책은?

✅ **답** (`maxmemory-policy`):

| 정책 | 의미 |
|---|---|
| `noeviction` | 메모리 가득 차면 쓰기 에러 (기본값) |
| `allkeys-lru` | 모든 키 중 가장 안 쓴 키 제거 |
| `allkeys-lfu` | 모든 키 중 가장 덜 자주 쓴 키 제거 |
| `volatile-lru` | **TTL 있는 키 중** LRU |
| `volatile-ttl` | TTL이 가장 짧게 남은 키 제거 |
| `allkeys-random` | 무작위 제거 |

**캐시 용도**: `allkeys-lru` 또는 `allkeys-lfu`.
**세션/락 용도**: `volatile-lru` (TTL 있는 캐시만 정리, 영구 키 보호).

### Q24. Redis 고가용성 구성 — Standalone / Sentinel / Cluster?

✅ **답**:

| 구성 | 특징 | 언제 |
|---|---|---|
| **Standalone** | 단일 인스턴스 | 학습·개발, 가용성 요구 낮음 |
| **Sentinel** | 마스터 + 복제본 + Sentinel(감시자) → 자동 failover | 가용성 필요, 데이터는 마스터 1대로 충분 |
| **Cluster** | 키를 16384 슬롯으로 샤딩, 노드 여러 대 | 데이터·트래픽 분산 필요 |

**주의**: Cluster에선 한 트랜잭션의 키들이 모두 같은 슬롯에 있어야 MULTI/EXEC·Lua 가능 → **Hash Tag** (`{user:42}:profile`, `{user:42}:sessions`) 사용.

---

### Q25. Redis 장애 시 애플리케이션은 어떻게 동작해야 하나?

✅ **답** (fail-open vs fail-closed):

- **fail-open** (Redis 죽어도 서비스 진행): 캐시·인증 코드처럼 "기능이 좀 망가져도 사용자가 가입은 해야" 하는 경우
- **fail-closed** (Redis 죽으면 거부): 분산락·결제 멱등성처럼 "안전이 더 중요한" 경우

```java
try { return redis.get(key); }
catch (RedisConnectionException e) {
    if (criticalForCorrectness) throw e;       // fail-closed
    else return loadFromDB();                  // fail-open
}
```

→ **선택 기준**: 데이터 정합성 vs UX. 인증 코드는 보통 fail-open, 결제 멱등성은 fail-closed.

### Q26. Redis 메모리 누수 진단을 어떻게 하나?

✅ **답** (3단계):

```bash
# 1. 전체 메모리 통계
> INFO memory

# 2. big key 탐지
$ redis-cli --bigkeys

# 3. 의심 패턴 키들의 TTL 확인 (가장 흔한 원인: TTL 누락)
$ redis-cli --scan --pattern 'cache:*' | head -100 | xargs -I {} redis-cli TTL {}
# -1 나오는 키들이 메모리 누수 후보
```

**가장 흔한 원인**: SET 시 EX 옵션 누락 → 영구 키로 쌓임.

## F. 함정·꼬리잡기

### Q27. TTL 있는 키를 옵션 없이 SET으로 덮어쓰면?

✅ **답**:

> **TTL이 사라지고 영구 키가 된다** (TTL → -1).
>
> 운영 사고 1순위. 캐시가 만료되지 않고 메모리에 쌓이게 됨.
>
> **해법**: `SET k v KEEPTTL` 또는 매번 EX를 같이 지정.

---

### Q28. `EXISTS k k k` 결과는?

✅ **답**:

> **3** (키가 있으면). EXISTS는 **중복 카운트**함.
>
> `EXISTS k1 k2 k3` 처럼 여러 키 동시 체크 시, 존재하는 키 개수 반환. 같은 키 3번 넣으면 3번 카운트.

---

### Q29. 큰 키를 `DEL`로 지우면 어떻게 되나?

✅ **답**:

> Redis가 **삭제하는 동안 다른 모든 명령이 블로킹**됨.
>
> 1000만 원소 Hash를 DEL 하면 수 초~수십 초 동안 Redis 전체가 멈출 수 있음.
>
> **해법**: `UNLINK` — 키를 즉시 "지워진 것처럼" 처리하고 실제 메모리 회수는 백그라운드.

---

### Q30. 운영 환경에서 `KEYS *` 를 치면?

✅ **답**:

> **그 동안 Redis 전체가 멈춘다** (단일 스레드).
>
> 키 1000만 개면 8~30초 멈춤. 모든 API 타임아웃. 큰 장애.
>
> **재발 방지**:
> ```
> rename-command KEYS ""
> rename-command FLUSHALL ""
> rename-command FLUSHDB ""
> ```
> `redis.conf` 에 추가해서 명령 자체를 비활성화.

> 💡 시나리오 10 참고.

---

### Q31. Redis에 `INCRBYFLOAT` 로 잔액 관리하면 왜 안 되는가?

✅ **답**:

> **IEEE 754 부동소수점 부정확성** 때문.
>
> `0.1 + 0.1 + 0.1 ≠ 0.3` — 누적되면 회계 오차 발생.
>
> **해법**: 원 단위 정수로 환산 후 `INCRBY` 사용. `1500.50원` → `150050` (전 단위).
>
> 추가: Redis는 어차피 잔액의 **마스터 저장소가 아님** — 항상 DB가 정답이고 Redis는 빠른 조회용으로만 써야 함.

## 🎯 면접 직전 30초 체크리스트

암기보단 흐름으로:

1. **Redis는 왜 빠른가** → 인메모리 + 단일 스레드 + 효율적 자료구조
2. **단일 스레드인데 왜 안 느린가** → 이벤트 루프, CPU가 병목 아님
3. **SET 옵션의 의미** → `NX/XX/EX/KEEPTTL/GET` 한 줄로 다 됨
4. **원자성이 깨질 수 있는 자리** → 두 명령 사이 = SETNX+EXPIRE, GET+DEL → Lua나 `SET ... NX EX`
5. **운영 금지 명령** → `KEYS *`, 큰 키 `DEL`, 옵션 없는 SET (TTL 날아감)
6. **분산락** → `SET NX EX` + UUID + Lua DEL, Redisson 추천
7. **고가용성** → Standalone / Sentinel / Cluster
8. **저장 전략** → RDB (스냅샷) + AOF (명령 로그)
9. **eviction** → `allkeys-lru` 가 캐시 표준
10. **Redis 장애 시** → fail-open vs fail-closed, 용도에 따라 선택

## 자주 나오는 후속 질문 패턴

면접관이 추가로 파고드는 단골 흐름:

```
Q. 분산락 어떻게 구현?
→ SET NX EX 답함
Q. 그럼 TTL 만료 전에 작업이 안 끝나면?
→ Redisson watchdog 또는 락 안에서 외부 호출 안 하기
Q. Redis가 다운되면 락이 다 풀려서 동시 접근 발생할텐데?
→ Redlock / 비즈니스 레벨 보호 (DB unique 제약) / 다층 방어
```

```
Q. 캐시 어떻게 구현?
→ Cache-Aside 답함
Q. 캐시와 DB 정합성은?
→ Write 시 캐시 무효화 (DEL), 또는 Write-Through
Q. 캐시 스탬피드는?
→ Jitter, 분산락, PER (Probabilistic Early Recomputation)
```

```
Q. Rate Limit 어떻게?
→ INCR + EXPIRE 답함
Q. Fixed Window 경계 문제는?
→ Sliding Window (Sorted Set)
Q. Sliding Window의 메모리 부담은?
→ Sliding Window Counter (Fixed + 보간) 또는 Token Bucket
```

→ **꼬리잡기 답을 미리 준비해두면 면접관이 "어, 이 친구 깊네" 하는 인상.**
