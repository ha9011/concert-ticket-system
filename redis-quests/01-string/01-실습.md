# String 실습 — 명령어 드릴 76선

> 직접 redis-cli에 한 줄씩 입력하며 결과를 확인하라.
> **답을 보지 말고** 먼저 추측 → 입력 → 결과 비교 → 헷갈리면 메모.

## 표기 규칙

- ★ ~ ★★★★★ : 난이도
- 🪤 : 함정 문제 (대부분 헷갈림)
- ⏱ : TTL 관련
- 🔢 : 원자성/숫자 연산
- ⚠ : 실무에서 자주 실수하는 포인트
- `>` : redis-cli 입력 프롬프트

## 사전 준비

```bash
# 깨끗한 상태에서 시작
> FLUSHDB        # ⚠ 현재 DB만 비움. 다른 데이터 있으면 주의!
> DBSIZE         # 0 인지 확인
```

> 💡 별도 터미널에서 `redis-cli MONITOR`를 켜두면 명령어 흐름이 다 보인다.

### Q54. ★★ 🪤 ⚠ 인자 순서 차이
다음 둘은 같은 결과지만 인자 순서가 다르다.
```
> SETEX k 30 "value"           ← (key, seconds, value)
> SET k "value" EX 30          ← (key, value, EX, seconds)
```
SETEX에서 second와 value 자리를 바꾸면?
(error) ERR value is not an integer or out of range

---

### Q55. ★★ SETNX의 한계
SETNX는 **TTL 옵션이 없다**. 락 만들 때 다음 두 방식 비교:
```
[방식 A — 위험]
> SETNX lock "uuid"
> EXPIRE lock 5             ← 사이에 죽으면? 데드락

[방식 B — 안전]
> SET lock "uuid" NX EX 5   ← 원자적
```
방식 A의 문제를 직접 시뮬레이션해보라 (SETNX만 하고 EXPIRE 안 함).
TNX의 한계는 "TTL을 따로 줘야 한다 = 따로 줄 수밖에 없는 순간이 존재한다 = 그 틈에 죽으면 데드락" 입니다. 그래서 Redis 2.6.12부터 SET ... NX EX가 추가됐고, 이게 분산락의 표준이 됐어요.

## §9. 함정 / 응용 / 종합

### Q67. ★★ ⚠ KEYS vs SCAN
```
> MSET user:1 a user:2 b user:3 c order:1 x order:2 y
> KEYS user:*              ← 운영에서 절대 쓰지 말 것 (블로킹)
> SCAN 0 MATCH user:* COUNT 100
```
`SCAN`은 커서 기반. 결과가 한 번에 안 나올 수도 있다 — 커서가 0이 될 때까지 반복해야 함.

---

### Q68. ★★ RENAME / RENAMENX
```
> SET old "x"
> RENAME old new
> EXISTS old
> GET new

> SET conflict "y"
> SET src "z"
> RENAMENX src conflict    ← dst가 있으면 실패
> RENAME src conflict       ← RENAME은 덮어씀
> GET conflict -> z
```

---

### Q69. ★★ COPY (Redis 6.2+)
```
> SET src "original"
> COPY src dst
> COPY src dst              ← 이미 있으면? 실패
> COPY src dst REPLACE
> SET src "modified"
> GET dst                   ← 깊은 복사? 얕은 참조? 깊은 복사
```

---

### Q70. ★★ DUMP / RESTORE
```
> SET k "hello"
> DUMP k                    ← 바이너리 시리얼라이즈
> (반환값 복사)
> RESTORE k2 0 "<dump바이너리>"
> GET k2
```
> ⚠ DUMP 결과는 그대로 redis-cli 인자로 못 넘기는 경우가 많다. 코드 레벨에서 더 자주 사용.

### Q74. ★★★ UNLINK vs DEL
```
> SET big "..." (큰 값)
> DEL big                  ← 동기 삭제 (큰 값이면 블로킹)
> SET big "..." 
> UNLINK big               ← 비동기 삭제 (백그라운드)
```
운영에서 큰 키 지울 때는 `UNLINK` 권장.

---

### Q75. ★★ EXPIRETIME / PEXPIRETIME (Redis 7+)
TTL은 남은 시간, EXPIRETIME은 만료 절대 시각.
```
> SET k "v" EX 100
> TTL k
> EXPIRETIME k             ← Unix seconds
> PEXPIRETIME k            ← Unix ms
```

### Q76. ★★★ 종합 미니 미션 — 인증 코드 플로우 재현
이메일 인증 코드 흐름을 redis-cli만으로 시뮬레이션.
```
[1. 코드 발급]
> SET auth:test@test "1234" EX 180 NX
> TTL auth:test@test -> 180
> GET auth:test@test -> 1234

[2. 같은 이메일로 또 발급 시도 (Rate Limit 시뮬)]
> SET auth:test@test "5678" EX 180 NX    ← 어떻게 동작? 180이 안지났으면 새로 생성하지 않음

[3. 사용자가 코드 입력 → 검증]
> GETDEL auth:test@test                   ← 원자적 조회+삭제
> GET auth:test@test                       ← nil 확인

[4. 만료 동작]
> SET auth:expire@test "9999" EX 3 NX
> (4초 후) GET auth:expire@test           ← nil
```
