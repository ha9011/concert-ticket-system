# 부록 — 멱등성(Idempotency) 실전 가이드

> 02-실무.md §3에서 짧게 다룬 멱등성 키 패턴의 **전체 흐름**을 깊게 정리.
> Stripe·토스페이먼츠·AWS API가 모두 따르는 표준 패턴.

---

## §1. 멱등성이란

### 정의

**같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 동일한 성질**.
수학에서 유래: `f(f(x)) = f(x)`.

### 일상 비유

| 동작 | 멱등? |
|---|---|
| 엘리베이터 버튼 한 번 vs 열 번 누름 | ✅ (한 번 옴) |
| 자판기 동전 투입 | ❌ (잔액 증가) |

### HTTP 메서드별

| 메서드 | 멱등성 | 설명 |
|---|---|---|
| `GET` | ✅ | 조회. 데이터 변화 X |
| `PUT` | ✅ | 전체 교체. 결과 동일 |
| `DELETE` | ✅ | 첫 요청에 삭제, 이후엔 이미 없음 |
| `POST` | ❌ | 호출마다 새 리소스 생성 |
| `PATCH` | ⚠️ | 구현에 따라 다름 |

→ **`POST`를 멱등하게 만드는 도구가 `Idempotency-Key`**.

### 왜 중요한가

1. **네트워크 재시도 안전성** — 타임아웃 후 클라이언트 재시도해도 중복 처리 X
2. **결제 시스템** — 중복 결제 방지 (Stripe, 토스 표준)
3. **메시지 큐** — at-least-once delivery에서 중복 처리 보호

## §2. Idempotency Key 핵심 패턴

### 문제 시나리오

```
사용자: [결제하기] 클릭
  ↓
POST /payments (10,000원)
  ↓
서버: 결제 처리 중... (3초)
  ↓
네트워크 끊김 💥 (사용자 응답 못 받음)
  ↓
사용자: "어? 안 됐나?" → 다시 클릭
  ↓
POST /payments → 또 결제 💸💸 20,000원
```

### 해결: 클라이언트가 키 발급, 서버는 키로 중복 판별

```http
POST /payments
Idempotency-Key: a3f9c1e2-7b4d-4f8a-9c1e-2b4d6f8a0c1e
Content-Type: application/json

{ "amount": 10000, "orderId": "ORDER-123" }
```

서버 동작:
```
1. Redis에서 "idem:payment:a3f9..." 조회
2. 없으면 → 결제 처리 → 결과 저장
3. 있으면 → 저장된 결과 그대로 반환 (결제 X)
```

## §3. 키는 언제 생성하는가 — **결제 시도 단위로 1번**

> "버튼 클릭마다"가 아니라 **"결제 의도 발생 시점"**.

### ❌ 잘못된 시점: 버튼 클릭 핸들러

```javascript
function handlePay() {
    const key = uuidv4();   // ❌ 클릭마다 새 키
    fetch('/payments', { headers: { 'Idempotency-Key': key } });
}
```

→ 두 번 클릭 = 두 개 키 = **중복 결제** 💥

### ✅ 올바른 시점: 결제 화면 렌더링 시 1번

```javascript
function PaymentPage() {
    // 컴포넌트 mount 시 딱 1번 생성
    const [idempotencyKey] = useState(() => uuidv4());

    const handlePay = () => {
        fetch('/payments', {
            headers: { 'Idempotency-Key': idempotencyKey },   // 항상 같은 키
            body: JSON.stringify({ amount: 10000 })
        });
    };
}
```

### 시점별 적합도

| 시점 | 적합도 | 비고 |
|---|---|---|
| 결제 화면 진입(mount) | ⭐⭐⭐⭐⭐ | 가장 일반적 |
| 주문서 작성 시작 | ⭐⭐⭐⭐ | 더 일찍 |
| 결제 다이얼로그 오픈 | ⭐⭐⭐⭐ | SPA 모달 결제 |
| **버튼 클릭 핸들러** | ❌ | 클릭마다 새 키 → 의미 X |
| **전역 변수 1개** | ❌ | 여러 결제가 키 공유 → 두 번째 결제 막힘 |

### 3가지 원칙

1. **같은 결제 시도 = 같은 키** (재시도 시 키 유지)
2. **다른 결제 시도 = 다른 키** (새 결제 시작 시 새 키)
3. **결제 의도가 시작되는 순간**에 생성

### ⚠️ 새로고침(F5) 함정

새로고침 = 컴포넌트 재마운트 → **새 키 생성** → 같은 주문이 두 번 결제 가능 😱

**해법**: `sessionStorage`에 키 보관

```javascript
const [idempotencyKey] = useState(() => {
    const stored = sessionStorage.getItem(`idem:${orderId}`);
    if (stored) return stored;
    const newKey = uuidv4();
    sessionStorage.setItem(`idem:${orderId}`, newKey);
    return newKey;
});
```

결제 완료 후엔 `sessionStorage.removeItem()`으로 정리.

### Stripe 공식 가이드

> "Generate the idempotency key **for each unique request, not for each retry**."
> → **"고유한 결제 시도마다 1개. 재시도엔 기존 키 재사용."**

## §4. 처리 상태 머신 — 단순 응답 캐싱의 빈틈 보완

### 단순 버전의 빈틈

```java
// ❌ 응답만 저장하는 단순 버전
String cached = redis.get(key);
if (cached != null) return cached;

// ... DB 결제 처리 (3초) ...

redis.setex(key, 86400, response);
return response;
```

**문제**: 첫 요청이 3초 처리하는 동안, 같은 키로 두 번째 요청이 오면 **둘 다 캐시 MISS** → 동시 결제!

```
10:00:00.000  요청A → MISS → 결제 처리 시작
10:00:00.500  요청B → MISS → 결제 처리 시작  ← 😱
10:00:03.000  요청A 완료 → 캐시 저장 💸
10:00:03.500  요청B 완료 → 캐시 덮어쓰기 💸💸
```

### 해결: 상태 머신 + `SET NX EX`

3가지 상태로 관리:

| 상태 | 의미 |
|---|---|
| `PROCESSING` | 처리 중 (응답 없음) |
| `SUCCESS` | 완료 (응답 같이 저장) |
| `FAILED` | 실패 (에러 응답 저장) |

상태 전이도:
```
[없음] --setIfAbsent--> [PROCESSING] --완료--> [SUCCESS]
                                    --실패--> [FAILED]
```

### 개선된 흐름

```
요청A → setIfAbsent(PROCESSING) → ✅ 선점 성공 → 결제 처리
요청B → setIfAbsent(PROCESSING) → ❌ 이미 있음 → 상태=PROCESSING
                                                → "처리 중" 응답 (409)
요청A 완료 → 상태=SUCCESS + 응답 저장
요청C → 상태=SUCCESS → 저장된 응답 그대로 반환 ✅
```

→ **결제 1번만**. 동시 요청도 안전.

### 핵심: `setIfAbsent`는 원자적 선점

```java
Boolean acquired = redis.setIfAbsent(
    redisKey,
    initialRecord,            // {status: PROCESSING, requestHash: ...}
    Duration.ofHours(24)
);
// 동시에 100개 요청 와도 정확히 1개만 true
```

## §5. 두 번째 요청에 응답하는 법 — **"실패"가 아니라 "재생"**

### 핵심 원칙

> 두 번째 요청은 거부하는 게 아니라 **첫 응답을 그대로 돌려준다**.
> 그래야 클라이언트가 "내가 N번 보냈는지" 신경 안 써도 됨.

### 상태별 응답 패턴

#### ① `SUCCESS` 상태 → 첫 응답 그대로 200 OK

```http
HTTP/1.1 200 OK
Content-Type: application/json
Idempotent-Replayed: true   ← 옵션: "이건 재생된 응답이야" 알림

{ "paymentId": "PAY-555", "status": "SUCCESS", "amount": 10000 }
```

#### ② `PROCESSING` 상태 → "잠시 후 재시도"

```http
HTTP/1.1 409 Conflict
Retry-After: 2   ← 표준 헤더, 2초 후 재시도

{ "code": "PAYMENT_IN_PROGRESS", "message": "결제 처리 중입니다" }
```

#### ③ `FAILED` 상태 → 첫 에러 응답 그대로

```http
HTTP/1.1 402 Payment Required
Idempotent-Replayed: true

{ "code": "CARD_DECLINED", "message": "카드가 거절되었습니다" }
```

#### ④ 같은 키 + 다른 body → 키 충돌 (422)

```http
HTTP/1.1 422 Unprocessable Entity

{ "code": "IDEMPOTENCY_KEY_CONFLICT", "message": "같은 키이지만 요청이 다릅니다" }
```

### 응답 코드 표

| 상황 | HTTP Status | Body | 헤더 |
|---|---|---|---|
| 첫 요청 → 성공 | 200 OK | 결제 결과 | - |
| 재시도 → 이미 성공 | **200 OK** | 첫 응답 그대로 | `Idempotent-Replayed: true` |
| 재시도 → 처리 중 | 409 | "처리 중" | `Retry-After: 2` |
| 재시도 → 이미 실패 | **첫 응답의 4xx** | 첫 에러 그대로 | `Idempotent-Replayed: true` |
| 같은 키, 다른 body | **422** | "키 충돌" | - |

### ❌ 하면 안 되는 응답

```
재시도에 409 + "DUPLICATE_REQUEST" 던지기
```

- 사용자 혼란: "결제 됐어 안 됐어?"
- 클라이언트 로직 복잡 (409를 "성공"으로 해석해야 함)
- 재시도 안전성 깨짐 (HTTP 라이브러리는 4xx를 실패로 간주)

## §6. Redis에 무엇을 저장하는가 — 4종 세트

### 저장 구조 (JSON)

```json
{
  "status": "SUCCESS",
  "httpStatus": 200,
  "responseBody": {
    "paymentId": "PAY-555",
    "status": "SUCCESS",
    "amount": 10000
  },
  "requestHash": "a3f9c1e2b4d8f0a5..."
}
```

| 항목 | 왜 필요? |
|---|---|
| ① **status** | PROCESSING / SUCCESS / FAILED 분기 |
| ② **httpStatus** | 실패였으면 402 등 원래 코드 재생 |
| ③ **responseBody** | 그대로 재생하려면 필수 |
| ④ **requestHash** | 같은 키 + 다른 body 충돌 감지 (422) |

### 저장 방식 2가지

#### A. JSON 한 덩어리 (실무 표준)

```java
String key = "idem:payment:" + idempotencyKey;
String json = objectMapper.writeValueAsString(record);
redis.setex(key, 86400, json);   // 24시간 TTL
```

✅ 간단, 한 번에 GET으로 다 가져옴
✅ 응답이 작으면(KB) 메모리 부담 없음

#### B. Redis Hash (큰 응답일 때)

```java
redis.hset(key, Map.of(
    "status", "SUCCESS",
    "httpStatus", "200",
    "responseBody", json,
    "requestHash", hash
));
redis.expire(key, Duration.ofHours(24));
```

✅ 상태만 빨리 확인 (`HGET status`) 효율적
❌ 약간 복잡

→ **실무 90%는 방식 A**.

### 메모리 계산

```
한 결제당 저장: 약 1~3 KB
하루 10만 건 → 약 200 MB (24시간 TTL이므로 항상 이 수준)
하루 1000만 건 → 약 20 GB → Redis 클러스터로 처리
```

### ⚠️ 저장 금지

| 항목 | 금지 이유 |
|---|---|
| 비밀번호, 카드번호 전체 | 보안. 절대 금지 |
| 요청 body 전체 (원본) | 해시만 (저장 효율 + 보안) |
| 응답이 큰 파일 (이미지) | S3 URL만 저장 |

## §7. requestHash — 해시 vs 암호화

### 해시는 단방향. 복호화 불가능.

```
[암호화] (양방향)
원본 ──암호화 키──→ 암호문
원본 ←─복호화 키── 암호문

[해시] (단방향)
원본 ──해시 함수──→ 해시값
원본 ✗ 해시값  (되돌릴 수 없음!)
```

### 비유

```
암호화 = 자물쇠 🔒  (열쇠 있으면 다시 열 수 있음)
해시   = 믹서기 🥤  (사과 → 사과주스. 주스를 사과로 못 되돌림)
```

### 해시의 4가지 특성

| 특성 | 의미 |
|---|---|
| **단방향** | 원본 → 해시 O, 해시 → 원본 ❌ |
| **결정적** | 같은 입력 → **항상 같은 해시** |
| **고정 길이** | 입력 1글자든 1GB든 출력 동일 (SHA256은 64자) |
| **충돌 거의 없음** | 다른 입력이 같은 해시일 확률 사실상 0 |

```java
sha256("amount=10000,orderId=ORDER-123")
  → "a3f9c1e2b4d8..."   (항상 64자)

sha256("amount=10000,orderId=ORDER-123")   // 다시 돌리면
  → "a3f9c1e2b4d8..."   // ✅ 같음! (결정적)

sha256("amount=10001,orderId=ORDER-123")   // 1만 다름
  → "f7b2d4e6..."        // 완전 다른 결과
```

### 왜 해시로 비교하나? — 3가지 이유

1. **비교 목적**: 같은 입력 → 같은 해시이므로 빠른 일치 확인
2. **메모리 절약**: 2KB body → 64byte 해시 (32배 절감)
3. **보안**: Redis 털려도 원본 카드번호/주소 등 노출 X

### 해시 vs 암호화 비교표

| 항목 | 해시 (Hash) | 암호화 (Encryption) |
|---|---|---|
| 방향 | **단방향** ⛔ | 양방향 ↔️ |
| 목적 | 무결성 확인, 비교 | 비밀 유지 |
| 길이 | 고정 (예: 64자) | 입력에 비례 |
| 예시 | SHA-256, MD5 | AES, RSA |
| 비밀번호 저장 | ✅ | ❌ (해킹 시 전부 노출) |
| 요청 body 비교 | ✅ | ❌ (오버스펙) |

### 자바 코드

```java
import java.security.MessageDigest;
import java.util.HexFormat;
import java.nio.charset.StandardCharsets;

public class HashUtil {
    public static String sha256(String input) {
        try {
            MessageDigest md = MessageDigest.getInstance("SHA-256");
            byte[] hash = md.digest(input.getBytes(StandardCharsets.UTF_8));
            return HexFormat.of().formatHex(hash);
        } catch (NoSuchAlgorithmException e) {
            throw new RuntimeException(e);
        }
    }
}

// 사용
String requestJson = objectMapper.writeValueAsString(req);
String hash = HashUtil.sha256(requestJson);
// → "a3f9c1e2b4d8..."
```

## §8. 풀 코드 (Spring 예시)

```java
@PostMapping("/payments")
public ResponseEntity<?> pay(
    @RequestHeader("Idempotency-Key") String idemKey,
    @RequestBody PaymentRequest req
) {
    String redisKey = "idem:payment:" + idemKey;
    String currentHash = HashUtil.sha256(toJson(req));

    String cached = redis.get(redisKey);

    if (cached == null) {
        // PROCESSING 상태 + 요청 해시 선점 저장
        IdempotencyRecord initial = IdempotencyRecord.builder()
            .status(PROCESSING)
            .requestHash(currentHash)
            .build();

        boolean acquired = redis.setIfAbsent(
            redisKey, toJson(initial), Duration.ofHours(24)
        );
        if (!acquired) {
            // 동시 요청이 방금 선점 → 처리 중 응답
            return ResponseEntity.status(409)
                .header("Retry-After", "2")
                .body(Map.of("code", "PAYMENT_IN_PROGRESS"));
        }

        try {
            PaymentResponse resp = paymentService.process(req);

            // SUCCESS 덮어쓰기
            IdempotencyRecord success = IdempotencyRecord.builder()
                .status(SUCCESS)
                .httpStatus(200)
                .responseBody(resp)
                .requestHash(currentHash)
                .build();
            redis.setex(redisKey, 86400, toJson(success));
            return ResponseEntity.ok(resp);

        } catch (PaymentException e) {
            // FAILED 덮어쓰기
            IdempotencyRecord failed = IdempotencyRecord.builder()
                .status(FAILED)
                .httpStatus(e.getHttpStatus())
                .errorResponse(e.toResponse())
                .requestHash(currentHash)
                .build();
            redis.setex(redisKey, 86400, toJson(failed));
            return ResponseEntity.status(e.getHttpStatus()).body(e.toResponse());
        }
    }

    // 이미 본 키 → 재생
    IdempotencyRecord record = fromJson(cached);

    // body 검증
    if (!record.getRequestHash().equals(currentHash)) {
        return ResponseEntity.status(422)
            .body(Map.of("code", "IDEMPOTENCY_KEY_CONFLICT"));
    }

    return switch (record.getStatus()) {
        case PROCESSING -> ResponseEntity.status(409)
            .header("Retry-After", "2")
            .body(Map.of("code", "PAYMENT_IN_PROGRESS"));
        case SUCCESS -> ResponseEntity.ok()
            .header("Idempotent-Replayed", "true")
            .body(record.getResponseBody());
        case FAILED -> ResponseEntity.status(record.getHttpStatus())
            .header("Idempotent-Replayed", "true")
            .body(record.getErrorResponse());
    };
}
```

## §9. 흔한 함정

| 함정 | 결과 | 해법 |
|---|---|---|
| 클라이언트가 키를 매 클릭마다 생성 | 멱등성 무용지물 | 결제 화면 mount 시 1회 생성 |
| 새로고침으로 키 사라짐 | 같은 주문 중복 결제 | sessionStorage 저장 |
| 응답만 캐싱 (PROCESSING 상태 없음) | 동시 요청 시 중복 처리 | 상태 머신 + setIfAbsent |
| 재시도에 4xx로 거부 | 클라이언트 혼란 | 첫 응답 재생 (200) |
| requestHash 없음 | 키 재사용 실수 못 잡음 | SHA256 해시 비교 |
| TTL 없음 또는 너무 짧음 | 메모리 누수 / 늦은 재시도 실패 | 24시간 표준 |
| 실패 시 키 즉시 삭제 | 재시도가 또 결제 시도 | FAILED 상태로 저장 (재생) |
| 비밀번호/카드번호 그대로 저장 | 보안 사고 | 해시만 저장 |

## §10. 헷갈리는 포인트 (FAQ)

### Q1. "결과 캐싱 안 함" 함정이 정확히 뭐예요?

**처리 사실은 기록했는데, 재시도 시 돌려줄 응답을 어디에도 준비 안 한 상태.**

```java
// ❌ 함정에 빠진 코드
redis.set(idemKey, "PROCESSED");   // 플래그만 박음
return resp;
```

```
1번째 요청 → 결제 성공 → 응답: { paymentId: "PAY-555", receiptUrl: "..." }
네트워크 끊김 💥
2번째 요청 → Redis에 "PROCESSED"만 있음 → 응답: "이미 처리됨" (빈 body)
       ↓
클라이언트: showSuccessPage(response.paymentId) → undefined! → 화면 깨짐 😱
```

→ 클라이언트는 paymentId·receiptUrl 같은 **결과 정보**가 있어야 화면을 그릴 수 있는데, 서버가 그걸 못 줌.

### Q2. 그럼 Redis 대신 DB에서 조회해서 응답 만들면 되지 않나?

**가능하지만, 권장하지 않는다.** 함정의 해법은 2가지:

| 방법 | 평가 |
|---|---|
| **A. Redis에 응답 캐싱** | ⭐ 실무 표준 |
| **B. DB에서 재조회해 응답 재구성** | 가능. 단, 함정 3개 있음 |

DB 조회로 대체할 때의 문제:

1. **DB 데이터는 시간 따라 바뀜** → 1번째 응답과 다를 수 있음 (환불 처리, 정책 변경 등) → 멱등성 정의 깨짐
2. **응답엔 DB에 없는 값도 있음** → PG사 일회성 토큰, 영수증 URL, 적립 정책에 따른 계산값 등
3. **DB 부하 50배 + 응답 50배 느림** → 재시도마다 무거운 쿼리

→ Redis 멱등성 캐시 = **"응답 박제 사진" 📸** (변하지 않음).
→ DB = **"살아있는 데이터"** (변할 수 있음). 둘은 역할이 다름.

### Q3. 24시간 지나면 영수증도 못 받는 거 아닌가?

**아니에요. 영수증은 받을 수 있어요. 다른 경로로.**

**재시도(retry) ≠ 조회(query)** 를 구분해야 함:

| 행위 | 사용하는 API | 사용하는 저장소 |
|---|---|---|
| 결제 요청·재시도 | `POST /payments` + Idempotency-Key | Redis (24h) |
| 결제 내역·영수증 조회 | `GET /payments/{id}` | **DB (영구)** |

```
시점 0:00       결제 → DB+Redis 저장
시점 0:01       재시도 → Redis HIT → 같은 응답
시점 25:00      Redis TTL 만료 → 키 삭제
시점 1주일 후   영수증 보고 싶음
               → /payments/PAY-555 호출 (다른 API)
               → DB 조회 → 영수증 PDF 받음 ✅
```

**Redis는 "단기 재시도용 메모장", DB는 "영구 기록 시스템".**

### Q4. 그래서 Idempotency Key의 진짜 역할은?

> **"같은 결제 요청을 두 번 처리하지 않기"** — 그것만.
>
> 영수증 보관, 결제 조회, 환불 처리 같은 건 멱등성 키와 무관한 다른 영역.

### Q5. 비유로 한 방에

```
영화관 매표소 🎬

[Redis = 직원 책상 메모장 📝 — 영업시간(24h)만 유효]
  "주문번호 ABC → 표 발권 완료, 자리 A12"
  같은 ABC 또 오면 "그때 그 표 그대로 드릴게요"
  영업 끝나면 메모장 폐기

[DB = 영구 발권 시스템 💾]
  모든 발권 영구 보관
  영수증 재발급, 환불, 조회는 다 여기서

[손님이 25시간 뒤 영수증 잃어버려서 옴]
  직원: "메모장엔 없어요. 마이페이지에서 영수증 받으세요" → DB 조회
```

### Q6. 핵심 통찰 — 4요소가 한 세트로 24시간 살아있어야 한다

| 요소 | 역할 |
|---|---|
| **Idempotency Key** | "내 요청 식별표" (재시도 인식) |
| **상태** (`PROCESSING`/`SUCCESS`/`FAILED`) | "지금 어디까지 처리됐어?" |
| **결과(응답 body)** | "재시도 시 돌려줄 데이터" ← 없으면 클라이언트가 깨짐 |
| **TTL 24시간** | 재시도가 일어날 수 있는 현실적 윈도우 |

→ **이 네 가지가 한 세트로 24시간 박제되어 있어야** 멱등성이 완성.

```
재시도가 가능한 동안 = 첫 응답이 박제되어 살아있어야 한다. 📸
그 박제 기간 = 24시간 = TTL.
그 이후 결과 보고 싶으면 → DB 조회(별도 API)로.
```

## §11. 이 프로젝트 적용 포인트

### 1. 이메일 큐 컨슈머 (현재)

`queue-server/src/main/java/com/concert/queue/worker/EmailMultiWorker.java` 의 메시지 처리에 dedup 키 추가:

```java
public void processEmail(EmailMessage msg) {
    String dedupKey = "dedup:email:" + msg.getId();
    if (!redis.setIfAbsent(dedupKey, "1", Duration.ofDays(1))) {
        log.info("Skip duplicate email: {}", msg.getId());
        return;
    }
    sendEmail(msg);
    emailLogRepository.save(...);
}
```

→ at-least-once 큐에서 중복 처리 방지 (간단 버전. 상태 머신까지 갈 필요는 없음).

### 2. 향후 결제 API

콘서트 티켓 결제 API 추가 시 §8 풀 코드 적용:
- 헤더 `Idempotency-Key` 필수
- 4종 세트(상태/httpStatus/응답/해시) 저장
- 24시간 TTL

### 3. 좌석 예매 API

좌석 예매도 결제와 동일 패턴. 사용자가 결제 버튼 더블 클릭하거나 새로고침해도 좌석 1개만 잡혀야 함.

## §12. 한 줄 정리

> **멱등성 = "같은 요청 N번 = 1번 처리 + N번 같은 응답"**
>
> 핵심 4가지:
> 1. **클라이언트가 결제 시도마다 키 생성** (mount 시 1회, sessionStorage 백업)
> 2. **서버는 PROCESSING → SUCCESS/FAILED 상태 머신**으로 동시성 방어
> 3. **재시도엔 첫 응답 그대로 재생** (4xx로 거부 ❌)
> 4. **Redis에 4종 세트 저장** (상태·httpStatus·응답·requestHash), 24시간 TTL
>
> 해시는 단방향(복호화 불가) → 비교 + 메모리 절약 + 보안 3박자.

## 외울 템플릿

### 클라이언트

```javascript
const [idempotencyKey] = useState(() => {
    const stored = sessionStorage.getItem(`idem:${orderId}`);
    if (stored) return stored;
    const newKey = uuidv4();
    sessionStorage.setItem(`idem:${orderId}`, newKey);
    return newKey;
});

fetch('/payments', {
    headers: { 'Idempotency-Key': idempotencyKey },
    body: JSON.stringify(req)
});
```

```java
// 1. PROCESSING 선점 (setIfAbsent)
// 2. 처리 후 SUCCESS/FAILED 덮어쓰기

```
