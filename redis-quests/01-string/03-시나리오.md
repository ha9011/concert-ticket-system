# 🎬 시나리오 1 ⭐ 보스급. 인증 코드 스팸 폭격

## 📋 상황

월요일 오전 10시, 슬랙에 알림이 떴다.

> **DevOps**: "SMS 비용 지난 1시간 사이에 47만원 청구됨. 트래픽 100배 폭증. 회원가입 페이지 인증 코드 발송 API임."

로그를 보니 같은 이메일 주소들로 1초에 100건씩 발송 요청이 들어오고 있다. 코드는 `AuthService.sendAuthCode()` 한 줄, 그대로 SMS API를 때린다.

**5분 뒤 회의 잡혔다. "어떻게 막을지" 가져가야 한다.**

## 🔒 제약

- 정상 사용자의 "재발송" UX는 살려야 한다 (너무 빡빡하면 CS 폭발)
- 분산 환경 — api-server 2대 운영 중
- DB 추가 부담 없이 **Redis만으로** 해결
- 배포는 30분 안에 가능해야 함

## ❓ 질문

어떤 방어층을 어떤 순서로 깔겠는가?

## 🛡️ 다층 방어 (Defense in Depth)

한 가지로는 못 막는다. 3~4개 층을 쌓는 게 정석.

### Layer 1: "유효한 코드 있으면 재발급 금지" (가장 강력, 가장 싸다)

```
SET auth:{email} {code} NX EX 180
└─ NX 실패 = 이미 발급된 코드가 살아있음 → "기존 코드를 확인해주세요"
```

**효과**: 같은 이메일은 **3분에 1번만** 발송. 한 줄 변경으로 99% 차단.
**현재 코드 변경 포인트**: `AuthService.java:39` 의 `set()` → `setIfAbsent()`.

### Layer 2: 이메일별 시간당 발송 횟수

```java
String key = "auth:rl:email:" + email;
Long count = redis.opsForValue().increment(key);
if (count == 1) redis.expire(key, 1, TimeUnit.HOURS);
if (count > 5) throw new TooManyRequestsException();
```

**효과**: Layer 1을 우회한 정상 재발송도 시간당 5회로 캡.

### Layer 3: IP별 분당 발송 횟수 (봇 차단)

```java
String key = "auth:rl:ip:" + clientIp + ":" + nowMinute();
Long count = redis.opsForValue().increment(key);
if (count == 1) redis.expire(key, 60, TimeUnit.SECONDS);
if (count > 10) throw new TooManyRequestsException();
```

**효과**: 한 IP에서 이메일을 바꿔가며 봇질하는 패턴 차단.

### Layer 4 (필요 시): CAPTCHA

위 3개로 모자라면 마지막에 reCAPTCHA. UX 비용이 큼.

## 🪤 면접관이 깊이 파고들면

### Q1. "IP가 NAT 뒤면 같은 IP에 정상 사용자가 여러 명 있을 수 있는데요?"

→ IP 제한은 분당 10회처럼 **느슨하게**. 이메일 제한이 본진. IP는 보조 방어.

### Q2. "Redis가 다운되면 인증 코드 발송도 다 막힐 텐데, 어떻게 처리하시겠어요?"

→ **fail-open vs fail-closed** 트레이드오프.
- fail-open: Redis 죽으면 모든 발송 허용 (UX 우선, 보안 약화)
- fail-closed: Redis 죽으면 모든 발송 거부 (보안 우선, UX 망함)
- 인증 코드는 **fail-open이 일반적** (사용자가 가입을 못 하는 게 더 큰 문제).
- 대신 Redis 헬스체크 + 알람 + Sentinel/Cluster로 가용성 확보.

### Q3. "Layer 2의 Fixed Window는 경계 직전에 폭주 가능하잖아요?"

→ **시나리오**: 10:59:59에 5번, 11:00:00에 5번 → 2초 만에 10번 발송.
→ **해법**: Sliding Window. String만으론 정확히 못 만들고 Sorted Set 필요. ([[redis-sorted-set]] 단계에서 다룸)
→ **실무 절충**: 대부분 인증 코드 수준에선 Fixed Window로 충분.

### Q4. "Layer 1이 그렇게 좋으면 Layer 2~3은 왜 필요해요?"

→ 공격자가 **이메일을 매번 바꿔서** 시도하면 Layer 1은 매번 통과. Layer 2(이메일별)도 통과. Layer 3(IP별)에서 막힘.

### Q5. "INCR + EXPIRE 두 줄로 나누면 원자성 깨지는 거 아니에요?"

→ 동시 요청 시 둘 다 1로 INCR되어 둘 다 EXPIRE 호출 — 큰 문제는 없음.
→ **완벽한 원자성을 원하면 Lua**:
```lua
local v = redis.call('INCR', KEYS[1])
if v == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
return v
```

---

## 🎯 핵심 takeaway

> **"막는 건 한 줄로 안 된다. 층층이 쌓아라. 가장 싼 방어를 가장 앞에 둬라."**
>
> - Layer 1 (`SET NX EX`) 은 비용 0, 효과 99% — 무조건 먼저
> - 모든 Layer는 Redis 명령어 1~2개로 구현됨
> - 인증 코드 보안 = **"`NX EX` 한 줄과 `INCR + EXPIRE` 패턴의 조합"**

</details>

# 🎬 시나리오 2. "어제까지 잘 되던 캐시가 오늘 다 사라졌어요"

## 📋 상황

월요일 오전, CS 팀에서 연락이 왔다. "사용자 프로필 페이지가 갑자기 느려졌어요. 매번 DB에서 다시 조회되는 느낌이에요."

코드를 봤더니 누군가 어제 이런 PR을 머지했다:

```java
public void updateProfile(Long userId, ProfileDto dto) {
    User user = userRepository.findById(userId).orElseThrow();
    user.update(dto);
    userRepository.save(user);

    // 캐시 갱신
    redisTemplate.opsForValue().set("user:" + userId, toJson(user));
}
```

문제가 뭘까?

## ❓ 질문

이 코드가 왜 캐시를 "다 사라지게" 만들었는가? 어떻게 고치겠나?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안

**문제**: `set()` 호출 시 **EX 옵션 없음** → 기존 TTL이 사라지고 영구 키가 됨.

잠깐, "영구 키"인데 왜 사라진다는 거지? 그 반대 아닌가?

→ 곰곰이 보면 **이 한 줄이 캐시를 "사라지게" 한 게 아니라, 캐시를 "영구로 만들어버린" 것**. CS 팀이 본 건 별개 문제(예: 캐시 키 변경, 무효화 누락 등)일 수 있음. **하지만 운영에서 더 자주 보는 진짜 문제는** 다음 코드 패턴:

```java
// 다른 곳에서 이런 코드가 있었다면
redisTemplate.opsForValue().set("user:" + userId, json, 10, TimeUnit.MINUTES);
// 그런데 이 PR이 머지된 뒤로는
redisTemplate.opsForValue().set("user:" + userId, json);   // TTL 없는 SET
```

→ 갱신 직후엔 캐시 잘 동작하지만, **이후 누군가 EXPIRE를 재호출하지 않는 한 영구 키로 메모리에 쌓임**.
→ 시간 지나면 메모리 누수, 그리고 "옛날 데이터를 영원히 들고 있는" 버그.

## ✅ 올바른 패턴 (3가지)

```java
// (1) 매번 TTL 다시 주기 (가장 흔함, 명시적)
redisTemplate.opsForValue().set(key, json, 10, TimeUnit.MINUTES);

// (2) KEEPTTL — 기존 TTL 유지
redisTemplate.execute((RedisCallback<Boolean>) c ->
    c.set(key.getBytes(), json.getBytes(),
          Expiration.keepTtl(), RedisStringCommands.SetOption.UPSERT));

// (3) 갱신 대신 무효화 — 다음 조회 때 다시 캐싱
redisTemplate.delete(key);
```

## 🪤 함정 한 줄

> **TTL 있는 키를 옵션 없이 SET으로 덮어쓰면 TTL이 -1(영구)로 날아간다.** 운영에서 "캐시가 만료 안 됨" 사고 1순위.

</details>

# 🎬 시나리오 3. 좌석 예매 oversell — 같은 자리 2명 결제됨

## 📋 상황

콘서트 티켓 오픈 1분 만에 CS 문의 100건. "같은 좌석 A-15 에 저랑 다른 사람 둘 다 결제됐어요."

DB를 확인하니 좌석 A-15에 대한 `Booking` 레코드가 진짜 2개 있다. 결제도 둘 다 성공.

코드는 이렇다:
```java
public void bookSeat(Long concertId, String seatNo, Long userId) {
    Seat seat = seatRepository.findByConcertAndNo(concertId, seatNo);
    if (seat.isBooked()) throw new AlreadyBookedException();
    seat.book(userId);
    seatRepository.save(seat);
}
```

## ❓ 질문

이 코드가 왜 같은 좌석에 2명 부킹을 허용하나? 어떻게 막을래?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안

**문제**: `findByConcertAndNo` → `isBooked()` 체크 → `save()` 사이에 다른 요청이 끼어들 수 있음 (TOCTOU race condition). api-server가 2대면 자바 `synchronized`도 무용지물.

## ✅ Redis 분산락 적용

```java
public void bookSeat(Long concertId, String seatNo, Long userId) {
    String lockKey = "lock:seat:" + concertId + ":" + seatNo;
    String owner = UUID.randomUUID().toString();

    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, owner, 10, TimeUnit.SECONDS);   // NX EX 10s

    if (!Boolean.TRUE.equals(acquired)) {
        throw new SeatLockedException();
    }

    try {
        Seat seat = seatRepository.findByConcertAndNo(concertId, seatNo);
        if (seat.isBooked()) throw new AlreadyBookedException();
        seat.book(userId);
        seatRepository.save(seat);
    } finally {
        // 반드시 owner 검증 후 해제 (Lua)
        redisTemplate.execute(releaseScript, List.of(lockKey), owner);
    }
}
```

**핵심 3가지**:
1. `NX EX 10s`: 락 획득과 TTL을 원자적으로
2. `owner = UUID`: 내가 잡은 락인지 식별 (DEL 시 검증)
3. Lua로 안전한 해제: TTL 만료 후 다른 클라이언트가 락 잡았을 때 내가 잘못 지우는 사고 방지

## 🪤 함정 한 줄

> **락 해제를 그냥 `DEL` 로 하면 다른 클라이언트의 락을 지운다.** 반드시 Lua로 owner 검증.

</details>

# 🎬 시나리오 4. 결제 콜백이 두 번 왔어요

## 📋 상황

PG사로부터 결제 완료 콜백이 한 주문에 대해 **2번** 들어왔다. 우리 서버는 두 번 다 처리해서 결과적으로 **포인트가 2번 적립**됐다. 사용자는 좋아했지만 회계팀이 분노 중.

PG사 답변: "네트워크 타임아웃 시 우리는 at-least-once로 재시도합니다. 멱등성은 가맹점 책임이에요."

## ❓ 질문

같은 콜백이 N번 와도 1번만 처리하려면?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안

PG사 콜백에는 거의 항상 **고유 트랜잭션 ID** 가 들어 있다. 그걸 멱등성 키로 사용.

```java
public void handlePaymentCallback(PaymentCallback cb) {
    String idemKey = "idem:payment:" + cb.getTxId();

    Boolean isFirst = redisTemplate.opsForValue()
        .setIfAbsent(idemKey, "PROCESSING", 24, TimeUnit.HOURS);

    if (!Boolean.TRUE.equals(isFirst)) {
        log.info("Duplicate callback ignored: {}", cb.getTxId());
        return;   // 멱등성: 두 번째부터 무시
    }

    try {
        processPayment(cb);   // 포인트 적립, 주문 상태 변경 등
        redisTemplate.opsForValue().set(idemKey, "DONE", 24, TimeUnit.HOURS);
    } catch (Exception e) {
        redisTemplate.delete(idemKey);   // 실패 시 재시도 허용
        throw e;
    }
}
```

**왜 24시간 TTL?**: 24시간 안에는 같은 콜백 재시도가 도착해도 차단. 그 이후는 메모리 회수.

## 🪤 함정 한 줄

> **처리 실패 시 멱등성 키를 안 지우면 영구 차단**. 재시도가 영영 막힘.

</details>

# 🎬 시나리오 5. 새벽 3시 DB 부하 폭증 — 캐시 스탬피드

## 📋 상황

매일 새벽 3시 정각에 DB CPU가 80%로 튀어오른다. 사용자 트래픽은 새벽인데도 왜?

조사해보니 인기 콘서트 목록 캐시 `cache:concerts:popular` 의 TTL이 정확히 1시간이라, **자정·1시·2시·3시 정각마다 동시 만료** 되고 있다. 만료 직후 수천 개 요청이 한꺼번에 DB로 몰려서 같은 쿼리를 동시에 수천 번 때린다.

## ❓ 질문

이 "캐시 스탬피드(Cache Stampede)" 어떻게 막을래?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안 (3가지 접근, 조합 가능)

### 1. TTL Jitter (가장 간단, 즉시 적용 가능)

```java
int baseSec = 3600;
int jitter = ThreadLocalRandom.current().nextInt(300);   // 0~5분 랜덤
redisTemplate.opsForValue().set(key, json, baseSec + jitter, TimeUnit.SECONDS);
```

→ 모든 키의 만료 시점이 흩어짐. 정각 폭주 사라짐.

### 2. 분산락으로 "DB 조회는 한 명만"

```java
String cached = redis.get(key);
if (cached != null) return cached;

String lockKey = "lock:cache:" + key;
Boolean ok = redis.setIfAbsent(lockKey, "1", 5, TimeUnit.SECONDS);
if (ok) {
    try {
        String data = queryDB();   // 나만 DB 호출
        redis.set(key, data, 1, TimeUnit.HOURS);
        return data;
    } finally { redis.delete(lockKey); }
} else {
    Thread.sleep(50);
    return redis.get(key);   // 다른 사람이 채워준 캐시 읽기
}
```

→ 캐시 미스 시 DB 호출이 1번만 일어나도록 강제.

### 3. PER (Probabilistic Early Recomputation)

만료 직전(예: TTL의 90%)부터 일부 요청이 미리 재계산. 만료 시점에 캐시가 이미 갱신돼 있도록.

→ 구현 복잡. Caffeine 같은 라이브러리가 이 패턴 지원.

## 🪤 함정 한 줄

> **인기 캐시일수록 TTL에 무작위 jitter 필수**. 정확한 1시간 TTL이 미덕이라는 환상을 버려라.

</details>

# 🎬 시나리오 6. 큐 컨슈머가 같은 이메일을 2번 보냈어요

## 📋 상황

`queue-server`의 `EmailMultiWorker`가 같은 이메일을 두 번 보낸 사용자가 보고됐다. 로그를 보니 워커 스레드 두 개가 같은 메시지를 처리했다.

원인: 디스패처가 `rightPop`으로 꺼낸 메시지를 워커에게 넘기는 사이에 워커가 실패→재큐잉→다시 처리됐다. 또는 워커가 처리 도중 죽고 다른 워커가 같은 작업을 잡았다.

큐 시스템의 본질: **at-least-once 전달** — 적어도 1번은 보장하지만 2번 이상도 가능.

## ❓ 질문

컨슈머 레벨에서 **"정확히 1번만 처리"** 보장하려면?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안 — dedup 키 (멱등성 키와 같은 발상)

**Step 1**: 메시지에 UUID 박기 (프로듀서 측, `AuthService.java:85`)
```java
String messageId = UUID.randomUUID().toString();
redisTemplate.opsForList().leftPush("mail:queue",
    new EmailMessage(messageId, email));
```

**Step 2**: 컨슈머에서 dedup
```java
public void processEmail(EmailMessage msg) {
    String dedupKey = "dedup:email:" + msg.getMessageId();

    Boolean isFirst = redisTemplate.opsForValue()
        .setIfAbsent(dedupKey, "1", 1, TimeUnit.DAYS);

    if (!Boolean.TRUE.equals(isFirst)) {
        log.info("Skip duplicate message: {}", msg.getMessageId());
        return;
    }

    sendEmail(msg.getEmail());
    emailLogRepository.save(...);
}
```

**왜 1일 TTL?**: 큐 재시도가 그렇게 늦게 도착하는 경우는 거의 없지만, 안전 마진. 메모리 부담은 작음.

## 🪤 함정 한 줄

> **at-least-once 큐 = 컨슈머 측 멱등성이 의무.** 큐가 정확히 1번 줄 거라고 가정하지 마라.

</details>

# 🎬 시나리오 7. 분산락 잡았는데 또 데드락?

## 📋 상황

분산락 적용 후 운영 중. 그런데도 "좌석이 락 걸려서 1시간째 안 풀려요" CS가 들어왔다.

원인 파악:
- 락 TTL: 5초
- 좌석 예매 로직 안에서 **외부 결제 API 호출** — 평소 1초지만 가끔 30초 걸림
- 외부 API가 30초 끌면 락 TTL은 진작에 만료
- 그동안 다른 클라이언트가 락 잡고 작업 진행
- 원래 클라이언트가 30초 만에 돌아와서 작업 완료 + 락 해제(Lua로 owner 검증 → 자기 락 아니라 해제 실패) ← 이건 다행
- 근데 더 큰 문제는 **같은 좌석에 두 명이 부킹 가능**해진 것 (락 의미가 사라짐)

## ❓ 질문

(a) 락 TTL을 늘리면? (b) Lua owner 검증이 막아주는 것과 못 막는 것은?

## 💡 답안

### (a) TTL 늘리기의 함정

TTL을 30초로 늘리면? → 평소 1초 작업도 30초 동안 락 점유. 동시성 망함.
TTL을 적당히 잡는 건 **추측 게임** — 안전한 해법이 아님.

### (b) Lua owner 검증의 한계

- ✅ 막아주는 것: "다른 클라이언트가 잡은 락을 내가 실수로 지우는 것"
- ❌ 못 막는 것: **TTL 만료 후 다른 클라이언트가 락을 잡아서 같은 자원에 동시 접근하는 것**

→ owner 검증은 "락 해제 안전성"만 보장. "임계 영역 안전성"은 보장 못 함.

### 진짜 해법

#### 1. 락 안에서 외부 API 호출하지 않기 (구조 변경)

```
[Before] 락 획득 → DB 조회 → 외부 결제 → DB 저장 → 락 해제
[After]  락 획득 → DB 좌석만 점유 → 락 해제 → 외부 결제 → DB 결제 결과 반영
```

→ 락 점유 시간을 가장 짧게.

#### 2. Redisson의 watchdog

Redisson `RLock`은 락 잡은 클라이언트가 살아있는 동안 **TTL을 백그라운드에서 자동 연장**.

```java
RLock lock = redisson.getLock("lock:seat:1:A1");
lock.lock(10, TimeUnit.SECONDS);   // 기본 watchdog 켜짐
try { /* 시간이 30초여도 락 유지됨 */ }
finally { lock.unlock(); }
```

→ 작업이 30초 걸려도 클라이언트가 살아있으면 락 유지. 클라이언트 죽으면 watchdog도 죽으니 자연스럽게 만료.

## 🪤 함정 한 줄

> **분산락 TTL은 "정답이 없는 추측" — Redisson watchdog 또는 짧은 락 + 구조 분리가 정석.**

# 🎬 시나리오 8. Redis 메모리가 1주일째 계속 늘어나요

## 📋 상황

Redis 메모리 그래프가 1주일째 우상향. maxmemory 80% 임박. 곧 eviction이 시작될 위기.

`redis-cli --bigkeys` 결과:
```
Biggest list  found 'mail:queue' has 0 items     ← 정상
Biggest hash  found 'user:session:abc' has 3 fields
Biggest string found 'cache:concert:1' is 142 KB  ← ??
```

키 패턴별 카운트:
```
auth:*          : 12,847 개   ← 인증 코드는 3분 TTL인데 왜 이렇게 많지?
cache:concert:* :    214 개
lock:*          :      3 개
```

## ❓ 질문

뭐가 의심되는가? 어떻게 조사할래?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안

### 의심 시나리오

1. **`auth:*` 키 12,847개 + 3분 TTL** = 분당 4000개 발급?
   → SMS 비용도 같이 폭증했을 것. 봇 공격 의심.
   → **`auth:*` 중 TTL이 -1인 키가 있는지 확인** (시나리오 2처럼 누군가 옵션 없이 SET 호출했을 수도).

2. **`cache:concert:1` 이 142 KB**
   → 콘서트 1개 캐시가 142 KB? big key. 메모리 압박 + 조회 시 네트워크 부담.
   → 구조 분해 (HASH로 필드별 분리) 또는 압축 필요.

### 조사 명령어

```bash
# TTL -1 키 찾기 (메모리 누수 원인 1순위)
> SCAN 0 MATCH auth:* COUNT 100
# 반환된 각 키에 대해
> TTL auth:xxx
# -1 나오면 메모리 누수 의심

# 키별 메모리 사용량
> MEMORY USAGE cache:concert:1

# 전체 메모리 통계
> INFO memory
```

### 조치

1. **TTL -1 키들 식별 → 코드의 SET 호출에서 TTL 누락한 곳 찾아 수정**
2. **big key는 UNLINK로 비동기 삭제** (DEL은 블로킹)
3. **maxmemory-policy 설정 확인** — `allkeys-lru` 같은 eviction 정책이 켜져 있어야 비상 시 자동 회수

## 🪤 함정 한 줄

> **메모리 누수의 99%는 "TTL을 줘야 할 키에 안 준 것"**. SET에 EX 빠진 곳을 찾아라.

</details>

# 🎬 시나리오 9. "5회 틀리면 잠금" 어떻게 만들죠

## 📋 상황

PM 요구사항: "사용자가 비밀번호를 5회 연속 틀리면 30분간 로그인 잠금. 분산 환경이고, 정확히 5회여야 함."

DB로 `failed_attempts` 컬럼? → 매 로그인 시도마다 UPDATE → 락 경합. 게다가 30분 후 자동 해제도 별도 스케줄러 필요.

## ❓ 질문

Redis로 어떻게 짤래?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안

```java
public void login(String email, String password) {
    String lockKey = "login:lock:" + email;
    String attemptKey = "login:attempts:" + email;

    // 1. 잠금 상태 확인
    if (Boolean.TRUE.equals(redisTemplate.hasKey(lockKey))) {
        Long ttl = redisTemplate.getExpire(lockKey, TimeUnit.SECONDS);
        throw new AccountLockedException(ttl);
    }

    // 2. 비번 검증
    if (!verifyPassword(email, password)) {
        Long count = redisTemplate.opsForValue().increment(attemptKey);
        if (count == 1) redisTemplate.expire(attemptKey, 30, TimeUnit.MINUTES);

        if (count >= 5) {
            redisTemplate.opsForValue().set(lockKey, "1", 30, TimeUnit.MINUTES);
            redisTemplate.delete(attemptKey);
            throw new AccountLockedException(1800L);
        }
        throw new InvalidPasswordException(5 - count);
    }

    // 3. 성공 시 카운터 초기화
    redisTemplate.delete(attemptKey);
}
```

**키 설계 핵심**:
- `login:attempts:{email}` — 실패 카운터, TTL 30분 (그 안에 안 틀리면 자동 리셋)
- `login:lock:{email}` — 잠금 플래그, TTL 30분 (자동 해제)
- 둘 다 TTL 기반 → **별도 스케줄러 불필요**

### 더 안전하게 (Lua)

INCR과 잠금 처리를 분리하면 동시 요청 시 6번째 시도가 잠금 직후 들어올 수 있음. Lua로 묶기:

```lua
local attempts = redis.call('INCR', KEYS[1])
if attempts == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
if tonumber(attempts) >= tonumber(ARGV[2]) then
    redis.call('SET', KEYS[2], '1', 'EX', ARGV[3])
    redis.call('DEL', KEYS[1])
    return 'LOCKED'
end
return attempts
```

## 🪤 함정 한 줄

> **"실패 카운터"는 TTL이 본체다.** TTL 없이 카운트만 올리면 영구히 누적되어 모든 사용자가 결국 잠긴다.

</details>

# 🎬 시나리오 10. KEYS 한 번 쳤다가 운영 멈춤

## 📋 상황

신입 개발자가 운영 Redis에 접속해서 디버깅 중이었다.

```
prod-redis:6379> KEYS user:*
```

그 순간부터 약 8초 동안 Redis가 응답하지 않았다. api-server의 모든 요청이 타임아웃. 알람 폭발. 신입은 패닉.

원인: Redis는 단일 스레드. `KEYS *` 같은 명령이 들어오면 그동안 다른 모든 명령이 대기. 키가 천만 개 있으면 8초든 30초든 걸린다.

## ❓ 질문

(a) 지금 당장 어떻게 복구? (b) 같은 일이 다시 일어나지 않게 하려면?

---

<details>
<summary>💭 펼치기</summary>

## 💡 답안

### (a) 지금 당장

- KEYS 실행 중인 클라이언트가 끝날 때까지 기다리는 것 외엔 방법 없음 (이미 진입한 명령은 중단 불가에 가까움).
- **KILL** 명령으로 클라이언트 강제 종료 가능하지만 KEYS 실행 자체를 중간에 끊지는 못함.
- 교훈: **운영에서 KEYS는 한 번이라도 치면 끝.**

### (b) 재발 방지

#### 1. KEYS / FLUSHALL / FLUSHDB 명령 비활성화

`redis.conf`:
```
rename-command KEYS ""
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command CONFIG ""
```

→ 명령 자체를 사용 불가로 만듦. 신입이든 누구든 절대 못 침.

#### 2. ACL로 권한 분리 (Redis 6+)

```
ACL SETUSER devops on >password ~* +@all -keys -flushall -flushdb
ACL SETUSER app    on >password ~user:* +get +set +del +expire
```

→ 애플리케이션 계정엔 KEYS 권한 자체를 안 줌.

#### 3. 진짜 키 검색이 필요하면

```bash
# SCAN을 항상 사용 (블로킹 없음, 점진적)
> SCAN 0 MATCH user:* COUNT 100
# 또는 redis-cli 헬퍼
$ redis-cli --scan --pattern 'user:*'
```

## 🪤 함정 한 줄

> **운영 Redis에 KEYS, FLUSHALL은 비활성화가 정답.** "조심해서 쓴다"는 약속은 깨진다.

</details>
