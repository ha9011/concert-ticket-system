# String 실습 — 개념 정리 & 실무 활용 요약

> 76문항 실습 후 가볍게 읽으며 머릿속 정리용. 명령어 문법보다 **"왜 이 명령어가 있고, 실무에서 어떻게 쓰이는가"**에 무게.

---

## §1. 기본기 — SET / GET / DEL / EXISTS / TYPE

### 문법 한눈에

| 명령어 | 동작 | 반환값 |
|---|---|---|
| `SET k v` | 값 저장 (덮어쓰기) | OK |
| `GET k` | 값 조회 | 값 or `nil` |
| `DEL k1 k2 ...` | 키 삭제 (여러 개) | 실제 지운 개수 |
| `EXISTS k1 k2 ...` | 존재 확인 | 존재 개수 (중복 카운트) |
| `TYPE k` | 자료구조 종류 | string / list / set / ... / none |

### 핵심 포인트

- **`DEL`은 idempotent** — 없는 키 지워도 에러 없이 `0` 반환. 안심하고 호출 가능.
- **`EXISTS`는 중복 키도 셈** — `EXISTS k k k`는 키가 있으면 3 반환.
- **빈 문자열도 값** — `SET k ""` 가능, `EXISTS=1`, `STRLEN=0`.
- **키 이름 거의 무제한** — 한글, 공백, 빈 문자열 다 OK. 단 관습은 `domain:id:field` (`user:42:name`).

### 실무 활용

```
[캐시 무효화]
- 사용자 정보 업데이트 시 → DEL user:42:profile
- 한 번에 여러 캐시 무효화 → DEL user:42:profile user:42:settings user:42:sessions

[Null 캐싱 함정 회피]
- "조회했는데 없음"을 빈 문자열 ""로 저장하면 GET이 "" 반환
- 코드에서 null vs "" 구분 안 하면 DB 재조회 무한루프 → 빈 문자열 캐싱은 신중하게
```

## §2. 문자열 조작 — STRLEN / APPEND / GETRANGE / SETRANGE

### 문법 한눈에

| 명령어 | 동작 | 비고 |
|---|---|---|
| `STRLEN k` | 바이트 길이 | 문자 수 ≠ 바이트 수 |
| `APPEND k v` | 뒤에 붙이기 | 키 없으면 새로 만듦, 반환값=길이 |
| `GETRANGE k start end` | 부분 문자열 | **양쪽 끝 포함**, 음수 인덱스 OK |
| `SETRANGE k offset v` | 부분 덮어쓰기 | offset이 크면 `\x00`으로 패딩 |

### 핵심 포인트

- **`STRLEN`은 바이트 수** — 한글 1자 = 3 byte (UTF-8). "한글" → 6.
- **`GETRANGE`는 inclusive** — `0..4`면 5글자. `start > end`면 빈 문자열 (에러 아님).
- **`SETRANGE`의 null padding** — `SETRANGE new 5 "X"`는 앞 5바이트를 `\x00`으로 채움. 거의 안 씀.

### 실무 활용

```
[APPEND - 누적 로그]
- 짧은 이벤트 스트림을 한 키에 이어붙이기
- 단점: 키 크기 무한 증가 → 보통은 List/Stream 쓰는 게 정석
- 쓸 만한 케이스: 짧은 세션 액션 트레일, 디버그용 누적

[STRLEN - 사이즈 체크]
- 캐시 값이 너무 커지지 않는지 사전 확인
- big key 모니터링용

[GETRANGE - 부분 조회로 트래픽 절약]
- 큰 JSON 캐시에서 일부만 잘라 보기 (드물지만 가능)
```

> 🚨 실무 솔직 평가: **APPEND/SETRANGE는 거의 안 씀**. 면접 함정 카드로만 기억해도 됨.

## §3. 숫자 연산 — INCR / DECR / INCRBY / INCRBYFLOAT 🔢

### 문법 한눈에

| 명령어 | 동작 | 원자성 |
|---|---|---|
| `INCR k` | +1 | ✅ |
| `DECR k` | -1 | ✅ |
| `INCRBY k n` | +n (정수) | ✅ |
| `DECRBY k n` | -n (정수) | ✅ |
| `INCRBYFLOAT k n` | ±n (부동소수점) | ✅ |

### 핵심 포인트 (함정 위주)

- **빈 키에 INCR 가능** — 자동으로 0부터 시작. SET 미리 안 해도 됨.
- **숫자 같은 문자열만 OK** — `"100원"`, `"100.5"`에 INCR → 에러.
- **Long.MAX_VALUE overflow** — `9223372036854775807`에 INCR → 에러.
- **부동소수점 정확도 함정** — `INCRBYFLOAT n 0.1` × 3 → 정확히 0.3 안 됨 (IEEE 754).
- **INCR과 INCRBYFLOAT 섞으면 망함** — `"10.5"`에 INCR → 에러 ("정수 아님").
- **후행 0 잘림** — `INCRBYFLOAT n 1.0` 결과는 `"4"` (`"4.0"` 아님).

### 실무 활용

```
[원자적 카운터의 진가]
- 분산 환경에서 락 없이 안전한 +1
- 자바에서 AtomicLong 쓰는 이유와 똑같은 동기

[페이지 조회수]
INCR views:post:42                  ← 무한히 안전, 동시에 100개 요청 와도 정확

[Rate Limit (분당 요청 수)]
INCR rl:user:42:202605221430        ← key에 분 단위 시각 박기
EXPIRE rl:user:42:202605221430 60   ← 1분 후 자동 폐기
- 카운트가 60 넘으면 → 차단
- 키 이름에 시간을 박는 게 핵심: 시간이 지나면 새 키로 자동 윈도우 교체

[재고 차감]
DECR stock:concert:1                ← 결과 ≥ 0 인지 체크
- 단, "음수 안 되게"는 INCR/DECR로 못 막음 → Lua 스크립트로 검사 필요

[가격/잔액에 INCRBYFLOAT?]
- ❌ 금융 데이터엔 절대 비추 (부동소수점 부정확)
- ✅ "원 단위 정수"로 저장하고 INCRBY 쓰기
- 잔액 1500원 → 1500 (원), 100원 차감 → DECRBY 100
```

> 💡 한 줄 결론: **INCR은 분산 환경의 동시성 카운터로 압도적으로 자주 씀.** 부동소수점은 가급적 피하고 정수로 환산.

## §4. TTL — EXPIRE / PERSIST / EXPIREAT ⏱

### 문법 한눈에

| 명령어 | 단위 | 비고 |
|---|---|---|
| `EXPIRE k 60` | 초 (상대) | 가장 흔함 |
| `PEXPIRE k 500` | ms (상대) | 짧은 TTL |
| `EXPIREAT k 9999999999` | Unix 초 (절대) | 정확한 시각 만료용 |
| `PEXPIREAT k ...` | Unix ms (절대) | |
| `TTL k` | 초 단위 남은 시간 | -1 (TTL 없음), -2 (키 없음) |
| `PTTL k` | ms 단위 | |
| `PERSIST k` | TTL 제거 (영구화) | |
| `EXPIRETIME k` | 만료 절대 시각 (초) | Redis 7+ |

### 핵심 포인트 — TTL의 함정들

- **`TTL` 반환값 의미** — 양수=남은 초, `-1`=TTL 없음, `-2`=키 없음.
- **🚨 SET 재호출로 TTL 날아감** — 가장 자주 나오는 운영 사고:
  ```
  SET k "v1" EX 60    ← TTL 60초
  SET k "v2"          ← 옵션 없이 덮어쓰면 TTL 날아감 (-1)
  ```
  해결책: `SET k "v2" KEEPTTL`
- **과거 시각 EXPIREAT** — `EXPIREAT k 100` (1970년) → 즉시 만료/삭제.
- **음수 EXPIRE** — `EXPIRE k -1` → 즉시 삭제.
- **Redis 7+ 옵션** — `EXPIRE k 50 GT/LT/XX/NX`:
  - `GT`: 기존 TTL보다 클 때만 (TTL 연장만 허용)
  - `LT`: 기존 TTL보다 작을 때만 (TTL 단축만 허용)
  - `XX`: TTL 있을 때만
  - `NX`: TTL 없을 때만 (영구 키만 대상)

### 실무 활용

```
[세션 관리]
SET session:abc123 "userdata" EX 1800    ← 30분 세션
- 사용자가 활동할 때마다 EXPIRE로 갱신 (sliding expiration)
- 또는 GETEX로 조회와 동시에 TTL 연장

[캐시 TTL]
SET cache:user:42 "{json}" EX 300        ← 5분 캐시
- TTL 안 주면 메모리 누수의 시작

[인증 코드]
SET auth:user@x.com "1234" EX 180 NX     ← 3분 + 중복 발급 방지
- NX가 핵심: 코드 유효한 동안 재발급 막아 SMS/이메일 비용 절감

[이벤트 시각 만료]
EXPIREAT event:concert:1 1735689600      ← "2025-01-01 09:00에 만료"
- 절대 시각이 필요할 때 (티켓 오픈, 쿠폰 만료 등)

[KEEPTTL 활용 - "값만 갱신, TTL 유지"]
SET cache:user:42 "{newjson}" KEEPTTL
- 5분 캐시 중 3분 지났을 때 값만 바꾸고 남은 2분은 유지
- 없으면 매번 새로 5분 늘어나서 캐시가 영영 안 만료될 수도
```

> 💡 **운영에서 가장 자주 깨지는 약속**: "값 업데이트할 때 TTL은 유지된다." → **TTL은 명시적으로 KEEPTTL을 안 주면 날아간다**. 무조건 기억.

## §5. SET 옵션 조합 — NX / XX / EX / GET / KEEPTTL

### 문법 한눈에

| 옵션 | 의미 |
|---|---|
| `NX` | 키가 **없을** 때만 SET |
| `XX` | 키가 **있을** 때만 SET |
| `EX 초` | TTL 초 단위 |
| `PX ms` | TTL ms 단위 |
| `EXAT 유닉스초` | 절대 시각 TTL |
| `KEEPTTL` | 기존 TTL 유지 |
| `GET` | SET 동시에 이전 값 반환 (Redis 6.2+) |

### 조합 규칙

- `NX`와 `XX`는 동시 불가 (syntax error).
- `EX 0`, `EX -1` 같은 비합리 TTL은 에러.
- 옵션 4중주 가능: `SET k "v" NX EX 60 GET` (조건부 + TTL + 이전 값까지 한 방).

### 실무 활용 — SET 옵션이 진짜 강한 곳

```
[분산락 (Distributed Lock)]
SET lock:seat:1 "uuid-A" NX EX 5
- NX: 이미 락 잡혀있으면 실패
- EX 5: 락 잡은 클라이언트가 죽어도 5초 후 자동 해제 (데드락 방지)
- "한 명령어로 락 획득 + TTL 설정" = SETNX+EXPIRE의 데드락 위험 회피

[중복 요청 차단 (멱등성 키)]
SET idempotency:txid-xyz "processed" NX EX 3600
- 같은 트랜잭션 ID로 들어온 2번째 요청은 NX 실패 → 무시
- 결제 콜백 중복 방지의 핵심

[조건부 갱신]
SET user:42:online "true" XX EX 60
- "온라인 상태인 유저"만 갱신 (없으면 안 만듦)
- 로그아웃한 유저 키를 실수로 부활시키는 것 방지

[atomic swap — 카운터 리셋과 값 회수]
SET hourly:requests "0" GET EX 3600
- 이전 1시간 카운트를 받고, 동시에 새 윈도우 시작
- GETSET의 대체재 (GETSET은 deprecated)
```

> 💡 SET의 옵션 조합은 **Redis가 단순한 KV가 아니라 "원자적 작업 도구"가 되는 지점**. 면접 단골 + 실무 빈출.

## §6. SETNX / SETEX / PSETEX — 옛 명령어들

### 한눈에

| 명령어 | 의미 | 현대 대체재 |
|---|---|---|
| `SETNX k v` | NX만 (TTL 없음) | `SET k v NX` |
| `SETEX k 30 v` | EX 30 | `SET k v EX 30` |
| `PSETEX k 500 v` | PX 500 | `SET k v PX 500` |

### 왜 지금은 거의 안 쓰는가

- **`SETNX`의 치명적 한계**: TTL 옵션이 없음 → `SETNX` + `EXPIRE`로 따로 줘야 함 → **두 명령 사이에 클라이언트가 죽으면 TTL 없는 락이 영원히 남음 (데드락)**.
- **`SETEX` 인자 순서가 헷갈림**: `(key, seconds, value)` — value와 seconds 순서 바뀌면 에러.
- 이 모든 한계 때문에 Redis 2.6.12부터 `SET ... NX EX ...` 통합 명령이 등장, 표준이 됨.

### 실무 활용

> **신규 코드: `SET ... NX EX`만 쓰면 됨.** SETNX/SETEX는 레거시 코드 읽을 때만 마주침.
> 면접에서 "왜 SETNX 대신 SET ... NX EX 쓰나요?" → **원자성 + 데드락 방지**.

## §7. 일괄 처리 — MSET / MGET / MSETNX

### 문법 한눈에

| 명령어 | 동작 | 원자성 |
|---|---|---|
| `MSET a 1 b 2 c 3` | 여러 키 한 번에 SET | ✅ (모두 성공) |
| `MGET a b c` | 여러 키 한 번에 GET, 순서 보장 | ✅ |
| `MSETNX a 1 b 2` | 모두 없을 때만 SET, 하나라도 있으면 전체 실패 | ✅ all-or-nothing |

### 핵심 포인트

- **MGET 응답 순서 = 요청 순서.** 없는 키는 `nil`.
- **MSET에는 TTL 옵션 없음** — 여러 키에 TTL 같이 주려면 `MULTI/EXEC` 또는 파이프라인.
- **MSETNX는 진짜 all-or-nothing** — 100개 키 중 1개라도 있으면 100개 다 안 들어감.

### 실무 활용

```
[N+1 round-trip 줄이기]
- 100명의 유저 이름 조회: 100번 GET → 100번 RTT
- MGET user:1:name user:2:name ... user:100:name → 1번 RTT
- 응답 속도 10~100배 차이 가능

[사용자 프로필 일괄 캐싱]
MSET user:42:name "ha" user:42:email "x@y" user:42:tier "gold"
- 단, "프로필을 키별로 흩어놓을지" vs "HASH로 묶을지"는 별개 결정
- 보통은 HASH 권장 (관련 데이터는 한 키에)

[MSETNX는 거의 안 씀]
- "모든 키 동시에 없을 때만" 같은 조건이 실무에 흔치 않음
- 면접용 지식으로만 기억
```

> 💡 **N+1 패턴이 보이면 MGET을 떠올려라**. Redis는 빠르지만 네트워크 RTT는 여전히 병목.

## §8. 원자적 GET 패턴 — GETSET / GETEX / GETDEL

### 문법 한눈에

| 명령어 | 동작 | 비고 |
|---|---|---|
| `GETSET k v` | 이전 값 반환 + 새 값 저장 | **deprecated** (`SET ... GET` 대체) |
| `GETEX k EX 30` | 조회 + TTL 갱신 | `PERSIST` 옵션도 있음 |
| `GETEX k` (옵션 X) | 그냥 GET처럼 동작 | |
| `GETDEL k` | 조회 + 즉시 삭제 | one-time token용 |

### 핵심 포인트

- **`GETSET` deprecated** — Redis 6.2부터 `SET k v GET`이 대체. 옵션 조합(NX/XX/EX) 가능해서 흡수됨.
- **`GETEX`는 "sliding session" 패턴의 핵심** — 조회와 동시에 TTL 연장.
- **`GETDEL`은 "한 번만 쓰고 버리는 토큰"용** — 단, **사용자가 타이핑하는 코드(OTP)엔 부적합** (오타 시 코드 날아감).
- **`GET`은 TTL을 건드리지 않음** — TTL은 계속 차감됨.

### 실무 활용

```
[GETEX - sliding session]
GETEX session:abc123 EX 1800
- 사용자가 API 호출할 때마다 세션 30분 연장
- "마지막 활동 30분 후 자동 로그아웃" 구현

[GETDEL - 안전한 곳]
GETDEL reset:password:abc123def456     ← 비밀번호 재설정 토큰 (URL의 긴 UUID)
GETDEL payment:callback:txid-xyz       ← 결제 콜백 idempotency 키
GETDEL oauth:state:randomstring        ← OAuth state/nonce
- 공통점: "사람이 타이핑하지 않음" → 오타 불가

[GETDEL - 부적합한 곳: OTP]
- ❌ GETDEL auth:user@x.com → 오타 시 코드 사라짐, 재발송 비용↑
- ✅ Lua 스크립트로 "compare-and-delete + 시도 횟수 카운터"
  - 일치 시에만 삭제
  - 틀리면 카운터 +1
  - 5회 틀리면 코드 무효화 (브루트포스 방지)

[SET ... GET - 카운터 리셋]
SET hourly:counter "0" GET EX 3600
- 이전 윈도우 값 받으면서 새 윈도우 시작
- 옛 GETSET 자리 대체
```

> 💡 **"읽으면서 뭔가 더 한다" 패턴 = 원자성이 필요한 자리**. 분리하면 race condition. 이 카테고리 명령어들이 그래서 존재.

## §9. 운영/관리 — KEYS / SCAN / RENAME / COPY / UNLINK

### 문법 한눈에

| 명령어 | 용도 | 주의 |
|---|---|---|
| `KEYS pattern` | 패턴 매칭 키 전체 | 🚨 **운영 절대 금지** (블로킹) |
| `SCAN 0 MATCH p COUNT 100` | 커서 기반 점진적 스캔 | 안전, 단 중복/누락 가능 |
| `RENAME old new` | 키 이름 변경 (덮어씀) | |
| `RENAMENX old new` | 대상 없을 때만 | |
| `COPY src dst [REPLACE]` | 깊은 복사 (Redis 6.2+) | |
| `DEL k` | 동기 삭제 (블로킹) | 큰 키엔 위험 |
| `UNLINK k` | **비동기 삭제** | 큰 키엔 이걸 |
| `OBJECT ENCODING k` | 내부 인코딩 (int/embstr/raw) | 메모리 튜닝용 |
| `OBJECT IDLETIME k` | 마지막 접근 후 경과 | LRU 디버깅용 |

### 핵심 포인트

- **`KEYS`는 운영에서 금지**: 100만 키 있으면 그동안 Redis 전체가 멈춤. dev에서만.
- **`SCAN`은 "필요한 시점이 곧 설계 점검 신호"**: 실시간 코드에서 SCAN이 필요하다면 SET/HASH로 인덱스를 별도로 두는 게 정석.
- **`UNLINK`는 big key 삭제의 정석**: 큰 List/Hash를 DEL 하면 그동안 다른 명령 다 막힘. UNLINK는 백그라운드 처리.
- **`WRONGTYPE` 에러**: Redis는 키 단위로 타입 고정. String 키에 LPUSH/SADD/ZADD 시도하면 에러. → 키 네이밍 컨벤션이 중요.

### 실무 활용

```
[SCAN이 정당한 곳 — 거의 다 "운영/배치"]
- 데이터 마이그레이션 (키 포맷 변경 시 1회성)
- 좀비 키 청소 배치
- 모니터링 도구 (redis-cli --bigkeys 등)
- 디버깅 ("user:* 키 몇 개야?")

[SCAN이 부적절한 곳]
- API 요청마다 SCAN → 응답 느려짐 + race condition
- 해결: 조회 패턴마다 SET/HASH 인덱스 따로 유지
  예) SADD user:123:sessions "abc" → SMEMBERS로 즉시 조회

[UNLINK의 효과]
- DEL big:cache:huge → 클라이언트 응답 지연 (수십 ms~수 초)
- UNLINK big:cache:huge → 즉시 응답, 백그라운드 정리

[OBJECT ENCODING - 메모리 최적화 단서]
- 작은 문자열 → embstr (44 byte 이하), 큰 문자열 → raw
- 정수만 들어가는 키 → int (별도 저장 없음, 매우 효율적)
- 면접에서 "메모리 절감" 질문 시 키워드로 활용
```

> 💡 **SCAN의 진짜 교훈**: "Redis는 검색 DB가 아니다. 조회 경로마다 자료구조를 따로 만든다."

## 🎯 String 실습 핵심 10개 패턴 (실무 빈출)

암기/체득해야 할 패턴만 추리면:

1. **`SET k v EX 초 NX`** — 분산락의 표준 형태
2. **`SET k v EX 초 NX`** — 멱등성 키 (중복 요청 차단)
3. **`INCR k` + `EXPIRE k 60`** — 슬라이딩 윈도우 Rate Limit
4. **`SET k v KEEPTTL`** — 값만 갱신, TTL 유지
5. **`GETEX k EX 초`** — sliding session (조회와 동시에 TTL 연장)
6. **`GETDEL k`** — one-time URL token (오타 가능한 OTP엔 부적합)
7. **`MGET ...`** — N+1 round-trip 제거
8. **`SET k v GET`** — atomic swap (카운터 리셋 + 값 회수)
9. **`UNLINK k`** — big key 안전 삭제
10. **Lua 스크립트** — "compare-and-delete + 카운터" (OTP 검증의 정답)

---

## ⚠️ 운영에서 절대 깨면 안 되는 약속 5가지

1. **`KEYS`는 운영에서 쓰지 않는다** — `SCAN`만 사용.
2. **TTL 있는 키를 옵션 없이 `SET`으로 덮어쓰지 않는다** — `KEEPTTL` 잊지 말 것.
3. **`SETNX` + `EXPIRE` 분리 호출하지 않는다** — `SET ... NX EX`로 원자화.
4. **big key를 `DEL`로 지우지 않는다** — `UNLINK` 사용.
5. **금융/잔액에 `INCRBYFLOAT` 쓰지 않는다** — 정수로 환산해서 `INCRBY`.

## 📌 면접에서 잘 나오는 비교 질문

| 질문 | 답 |
|---|---|
| `SETNX` vs `SET NX EX` | 원자성. SETNX+EXPIRE는 사이에 죽으면 데드락. |
| `GETSET` vs `SET ... GET` | GETSET deprecated. SET...GET은 옵션 조합 가능. |
| `DEL` vs `UNLINK` | DEL 동기/블로킹, UNLINK 비동기. big key는 UNLINK. |
| `KEYS` vs `SCAN` | KEYS 블로킹, SCAN 커서 기반. SCAN도 hot path엔 부적합. |
| `EXPIRE` vs `EXPIREAT` | 상대 시간 vs 절대 시각. |
| TTL `-1` vs `-2` | -1: TTL 없는 영구 키. -2: 키 자체 없음. |
